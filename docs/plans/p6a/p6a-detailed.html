1. 개요

Goal

데스크톱이 모바일 디바이스를 신뢰 목록(device registry)에 올리는 입구를 만든다. 사용자는 QR 원샷 또는 6자리 코드(5분 만료) 두 경로로 디바이스를 페어링하고, 데스크톱은 디바이스별 토큰을 발급해 Keychain에 보관한다. 발급 이후 모든 WS 연결은 Authorization: Bearer를 요구하며, 미인증 연결은 즉시 close된다. 모바일 클라이언트 없이 DaemonDevCLI 확장으로 디바이스를 흉내내어 토큰 등록 → WS 인증 라운드트립을 e2e 검증한다.

Side

desktop (macOS, Apple Silicon). 모바일 트랙은 P8 이후이며, 본 단계는 데스크톱 단독으로 완결된 vertical slice를 가진다.

2. RALPLAN-DR Summary

2.1 Principles (설계 원칙)

  1. 신뢰 경계는 단일 게이트로 — WS 인증은 WSAuthGate 한 곳에 격리한다. 게이트 ②의 nonce echo 매칭 후 secret 대조가 같은 검증 로직(DeviceTokenVerifier)을 호출하여 인증의 실질(secret 대조)이 한 곳에 모인다. 게이트 ①은 구조 검증·nonce 키 등록만 하고 secret 대조를 게이트 ②로 지연한다.
  2. 비밀은 Keychain 전용, 테스트는 fake로 — 실 토큰 secret은 KeychainDeviceStore를 통해서만 디스크에 닿는다. 단위 테스트는 InMemoryDeviceStore로 entitlement/서명 오염 없이 정책을 검증하고, 실 Keychain은 GUI/CLI 검증 게이트로 분리한다.
  3. 토큰은 로그·운영 envelope payload에 절대 평문 노출 금지 — deviceTokenSecret은 (1) QR 인코딩, (2) pairing.response 본문, (3) Authorization 헤더, 세 곳에만 존재한다. input/output/ack/error 등 운영 envelope의 payload에는 절대 담지 않는다. pairing.response가 secret을 담는 것은 명시적 예외다 — 페어링 전용 kind, loopback 한정, 일회성(claim 1회 소비)이라 운영 채널과 신뢰 모델이 다르다. 로그는 토큰 ID(앞 8자 + 마스킹)만 남긴다. 이는 grep 검증 가능한 종료 조건이다.
  4. 두 페어링 경로는 평등하게 — QR과 6자리 코드는 같은 PairingPayload를 다른 전달 채널로 옮길 뿐, 발급된 토큰의 권한·수명은 동일하다. 어느 경로든 같은 DeviceStore.upsert로 수렴한다.
  5. P6b/P7을 수용하는 스키마, 미배선 정책 — Device.expiresAt/revoked/fcmToken/apnsToken 필드는 P6a에 두되 lifecycle UI·실 push는 미배선이다. ADR-A는 P7 freeze 전 동결할 정책만 확정하고 코드 강제는 해당 phase로 넘긴다.

2.2 Decision Drivers (Top 3)

  1. 핸드셰이크 reject의 실증 불확실성 — setClientRequestHandler가 connection 객체를 받지 않아 "이 연결 = 어느 디바이스"를 handler만으로 못 묶는다(그래서 게이트 ②를 carry-over 일치 증명으로 둔다). 또한 reject 시 클라이언트 관측 동작(HTTP 상태/즉시 close)이 SDK 문서에 빈약하다. Day 0 스파이크가 실패하면 게이트 전략(2단 → 첫-envelope 단일, 옵션 B)을 전환해야 하므로 단계 성패의 1차 분기다.
  2. 보안 결함 0 허용 (deliberate) — timing 공격, 토큰 평문 누출, 핸드셰이크 우회는 모두 "동작은 하지만 안전하지 않은" 결함이다. 기능 테스트로는 잡히지 않으므로 constant-time 검증·로그 마스킹·우회 시나리오 테스트를 1급 종료 조건으로 둔다.
  3. 기존 WS 자산 비파괴 마이그레이션 — DaemonDevCLI 4모드 + RoundTripTests/DaemonSmokeTests가 헤더 미첨부 클라이언트로 green이다. Bearer 의무화는 이들을 전부 깨뜨린다. 마이그레이션 범위를 Day 2에 명시하고 336 기준선 비파괴를 검산한다.

선택 근거 — Option A (carry-over 증명)

SDK 제약(handler가 connection 객체 미수신)이 옵션 C의 identity 보증을 구조적으로 막는다 — 핸드셰이크에서 검증한 헤더와 실제 연결을 묶을 다리가 없다. 옵션 B는 master의 "미인증 즉시 close"를 핸드셰이크가 아닌 첫 프레임 후로 늦춘다. 옵션 A는 핸드셰이크 헤더 Bearer를 구조 검증해 클라이언트의 일회성 nonce를 키로 (tokenId, secret)를 pendingAuth에 등록(coarse 차단 + 격리점 선확보) + 첫 envelope이 echo한 nonce와 일치하는 등록 항목만 소비하고 그 secret을 대조해야 승격하는 nonce 매칭 carry-over 일치 증명(fine 바인딩)으로 두 약점을 메운다. 토큰·secret은 헤더로만 운반되고 첫 envelope에는 다시 싣지 않으며 nonce만 echo한다. envelope의 self-asserted actor.deviceId가 아니라 "방금 핸드셰이크를 통과한 그 토큰을, 같은 nonce를 아는 첫 화자가 가져왔다는 carry-over 사실"이 identity를 묶으므로, 유효 토큰 탈취자가 deviceId를 위조해도 통과할 수 없다(secret 대조 + replay 1회 소비). nonce 키 매칭은 서로 다른 토큰의 동시 핸드셰이크에서도 각 connection이 자신의 항목만 소비하게 해 identity 교차 바인딩을 차단한다. secret 대조는 게이트 ①(핸드셰이크 큐)이 아니라 게이트 ②(actor 컨텍스트)에서 일어나 동일 DeviceTokenVerifier 한 곳으로 수렴한다. 단, Day 0 스파이크로 reject 클라이언트 관측을 실증한 뒤에만 A를 확정한다 — 스파이크 실패 시 B로 폴백하되, 인증의 실질(secret 대조)은 게이트 ②에 있으므로 보안 강도는 보존된다.

5.2 DeviceTokenVerifier — constant-time (보안 핵심)

// 왜 constant-time인가: 토큰 비교가 첫 불일치 바이트에서 early-return하면
// 응답 시간이 일치 prefix 길이에 비례해 누출된다(timing 공격으로 토큰 추측 가능).
// 모든 바이트를 항상 비교하고 누적 XOR로만 판정한다. pre-mortem 시나리오 1 방어.
public struct DeviceTokenVerifier: Sendable {
    let store: any DeviceStore

    // 게이트 ②(actor 컨텍스트)가 호출하는 단일 검증점. 입력은 carry-over가 넘긴 tokenId +
    // 핸드셰이크 헤더에서 추출돼 pendingAuth에 함께 carry된 presented secret이다.
    // Keychain 조회(store.secret)는 여기(actor)서 일어난다 — 핸드셰이크 큐를 블록하지 않는다.
    public func verify(tokenId: String, presentedSecret: Data) async -> VerifiedDevice? {
        guard let stored = try? await store.secret(forTokenId: tokenId) else { return nil }
        // CryptoKit 또는 수동 누적 XOR. 길이 불일치도 상수시간 처리(패딩 비교).
        guard Self.constantTimeEqual(presentedSecret, stored) else { return nil }
        guard let device = try? await store.find(byTokenId: tokenId),
              !device.revoked, device.expiresAt > Date() else { return nil }
        return VerifiedDevice(deviceId: device.id, tokenId: tokenId)
    }

    // 누적 차이만 본다 — 첫 불일치에서 멈추지 않아 시간이 입력에 비의존.
    static func constantTimeEqual(_ a: Data, _ b: Data) -> Bool {
        var diff = UInt8(a.count ^ b.count)
        let n = max(a.count, b.count)
        for i in 0..<n {
            diff |= (i < a.count ? a[i] : 0) ^ (i < b.count ? b[i] : 0)
        }
        return diff == 0
    }
}

5.3 WSServer 인증 2단 게이트 배선 (D-1 옵션 A — carry-over 증명)

핵심 메커니즘 — carry-over 일치 증명 (헤더/사실-기반)

setClientRequestHandler 클로저는 connection 객체를 받지 않는다. 따라서 핸드셰이크에서 본 Bearer를 newConnectionHandler→accept가 발급하는 clientId와 직접 묶을 다리가 없다(connectionBearer[clientId] 같은 매핑은 채워질 수 없다). 게이트 ②는 envelope이 self-assert한 actor.deviceId를 검증해서는 안 된다 — 유효 토큰 탈취자가 그 토큰이 가리키는 deviceId를 그대로 적으면 통과하는 동어반복이기 때문이다. 대신 게이트 ②를 nonce 매칭 carry-over 일치 증명으로 정의한다: 토큰·secret은 WS 핸드셰이크 additionalHeaders로만 운반되고 첫 envelope에는 다시 싣지 않으며, 클라이언트가 연결마다 생성한 일회성 nonce만 X-Pair-Nonce 헤더로 함께 보내고 첫 envelope이 그 nonce를 echo한다(nonce는 토큰·secret이 아닌 공개값이라 운영 payload에 토큰을 절대 두지 않는 Principle 3과 무관). 핸드셰이크 클로저(게이트 ①)가 헤더 Bearer를 구조 분해해 (tokenId, presentedSecret)를 nonce를 키로 pendingAuth에 등록한다(secret 대조는 핸드셰이크 큐를 블록하므로 하지 않고 carry만). 게이트 ②는 첫 envelope이 echo한 nonce가 핸드셰이크 등록 항목과 일치함을 증명하여(nonce 일치로 그 connection의 항목만 소비) 소비된 presentedSecret을 그 tokenId의 Keychain secret과 actor 컨텍스트에서 constant-time 대조해 승격한다. 즉 증명은 "envelope이 토큰을 또 제시함"이 아니라 "방금 시간창 내 핸드셰이크를 통과한 그 토큰·secret을, 같은 nonce를 아는 첫 화자가 가져왔다는 carry-over 사실"이다.

첫 envelope 토큰 운반 규약 (단일 매체 + nonce echo)

토큰·secret의 운반 매체는 WS 핸드셰이크 additionalHeaders의 Authorization: Bearer 한 곳뿐이다. 첫 envelope의 바이트에는 토큰도, 별도 토큰 필드도 없고 핸드셰이크의 X-Pair-Nonce를 echo하는 nonce 하나만 담긴다(WSEnvelope 스키마 무변경 — payload 내부 값일 뿐, nonce는 토큰·secret이 아닌 일회성 공개값이라 Principle 3과 무관). 게이트 ②가 tokenId를 envelope에서 추출하지 않는 이유가 이것이다 — 헤더로 받은 토큰은 envelope 본문에 존재하지 않기 때문이다. 핸드셰이크 시점(게이트 ①)에 헤더에서 추출·nonce를 키로 pendingAuth 등록을 끝내고, 게이트 ②는 첫 envelope이 echo한 nonce로 그 항목만 지목해 소비할 뿐이다. (대안인 "envelope 신규 토큰 필드"는 §5.5의 pairing.*처럼 스키마·Principle 3 예외를 새로 열어야 해 채택하지 않는다 — 헤더 단일 매체 + nonce echo가 운영 payload 토큰 0 원칙과 자연 정합.) 옵션 B 폴백에도 동일 운반 규약이 적용된다: 핸드셰이크 클로저는 reject 없이 전부 accept하되 헤더 Bearer를 여전히 분해해 (tokenId, presentedSecret)를 nonce 키로 pendingAuth에 carry하고 첫 envelope의 nonce echo로 소비한다(SDK가 게이트 ②에 connection별 헤더를 안 주는 제약은 옵션 B에서도 같으므로 nonce carry는 필수). 게이트 ②의 nonce 매칭 소비·secret 대조 경로는 옵션 A와 동일하다 — 옵션 B는 "구조 검증 reject를 생략"할 뿐 carry-over 자체는 유지된다.

// 게이트 ① — 핸드셰이크. setClientRequestHandler는 connection 객체를 안 주므로
// 여기서는 토큰의 "구조적 유효성"만 본다(coarse). secret 대조는 게이트 ②로 지연한다 —
// 핸드셰이크 큐에서 Keychain 동기 조회를 하면 accept 큐가 블록되기 때문이다.
// 클라이언트가 함께 보낸 일회성 nonce를 키로 pendingAuth에 등록해, 게이트 ②가 '바로 그
// connection'의 항목만 nonce 일치로 소비하게 한다(FIFO cross-소비로 인한 identity 교차 차단).
// makeListenerParameters 안에서 ws 옵션에 부착(actor 외부 — @Sendable 클로저).
private static func makeListenerParameters(authGate: WSAuthGate) -> NWParameters {
    let params = NWParameters.tcp
    let ws = NWProtocolWebSocket.Options()
    ws.autoReplyPing = true
    ws.setClientRequestHandler(authGate.queue) { _, headers in
        // headers: [(name, value)]. Authorization 헤더에서 Bearer(tokenId.secret) 추출 +
        // X-Pair-Nonce 헤더에서 연결마다 신규 무작위 nonce(클라이언트 생성) 추출.
        // nonce 형식: SecRandomCopyBytes 16바이트 무작위값의 base64url 인코딩. isNonceRegistered
        // 충돌(reject) 시 클라이언트는 새 nonce를 생성해 재시도한다(충돌 확률은 2^128분의 1 수준).
        // 구조 검증만(tokenId 형식 + secret base64url 형식 + nonce 형식). Keychain 조회·secret 대조는
        // 여기서 안 한다 — 핸드셰이크 큐를 블록하면 안 되므로 대조는 게이트 ②(actor)로 지연한다.
        guard let (tokenId, presentedSecret) = authGate.structurallySplit(Self.bearer(from: headers)),
              let nonce = Self.nonce(from: headers),
              !authGate.isNonceRegistered(nonce) else {   // 중복 nonce는 reject
            return NWProtocolWebSocket.Response(status: .reject, subprotocol: nil)
        }
        // nonce 키로 carry-over 등록: (nonce -> (tokenId, presentedSecret, 등록시각)).
        // 토큰/secret은 여기(헤더)서만 보고 첫 envelope에는 다시 싣지 않는다(단일 매체).
        // nonce만 첫 envelope이 echo해 '어느 핸드셰이크가 이 연결인지'를 지목한다.
        authGate.registerPending(nonce: nonce, tokenId: tokenId, secret: presentedSecret, at: Date())
        return NWProtocolWebSocket.Response(status: .accept, subprotocol: nil)
    }
    params.defaultProtocolStack.applicationProtocols.insert(ws, at: 0)
    params.requiredLocalEndpoint = NWEndpoint.hostPort(host: "127.0.0.1", port: .any)
    return params
}

// 게이트 ② — 첫 envelope. carry-over 일치 증명 + secret 대조.
// 순서가 핵심: decode → 게이트 ②(인증) → ingestInbound(seq 전진) → kind switch(bind).
// 인증 미통과 시 ingestInbound/bind에 절대 진입하지 않는다 — 미인증 연결이 registry seq나
// 세션 바인딩(PTY 부수효과)을 오염시키지 못하게 한다.
private func handleMessage(_ data: Data, clientId: UUID, connection: NWConnection) async {
    let env: WSEnvelope
    do { env = try EnvelopeCodec.decode(data) }
    catch { /* 기존 malformedSeq/malformedPayload 에러 응답 (변경 없음) */ return }

    // 게이트 ② — ingestInbound(seq) 이전에 인증. 첫 envelope이 echo한 nonce가 트리거.
    if authState[clientId] == nil {
        // 토큰·secret은 envelope에 없다(헤더로만 운반). env.actor.deviceId도 신뢰 입력 아님.
        // 첫 envelope payload는 핸드셰이크에서 보낸 nonce만 echo한다(토큰·secret이 아닌 일회성 공개값
        // — payload echo는 Principle 3(운영 payload 토큰 금지)과 무관). 미echo·미등록 nonce는 거부.
        // echo 위치: 첫 envelope(session.start)의 payload JSON에 "nonce" 필드를 합류(예:
        // {"sessionId":"...","nonce":"..."}). 다른 kind가 먼저 도착하면 nonce 미echo로 간주해 UNAUTHORIZED.
        // consumePending(nonce:): 그 nonce로 등록된 pendingAuth 항목만 원자 소비 — '바로 그 connection'.
        guard let echoedNonce = Self.echoedNonce(env.payload),
              let claimed = await authGate.consumePending(nonce: echoedNonce, within: pendingWindow),   // -> (tokenId, presentedSecret)?
              let verified = await verifier.verify(tokenId: claimed.tokenId,
                                                  presentedSecret: claimed.secret) else {   // constant-time 대조(Keychain은 actor서 조회)
            send(makeError(code: DaemonErrorCode.unauthorized.rawValue,
                          message: "unauthenticated connection"), to: connection, clientId: clientId)
            connection.cancel()   // master "미인증 즉시 close" — registry 미오염
            return
        }
        authState[clientId] = verified   // 승격: 이후 envelope은 통과(carry-over 1회로 충분)
    }

    // 인증 통과 후에만 seq 전진 + kind 처리. 순서 역전 금지.
    do { try await registry.ingestInbound(clientId: clientId, env: env) }
    catch { /* 기존 NON_MONOTONIC_SEQ 에러 응답 (변경 없음) */ return }
    // ... 기존 kind switch (.sessionStart → bind + sessionStartHandler 등) ...
}
carry-over 시간창 만료 정책

pendingAuth 항목은 pendingWindow = 10초 내에 첫 envelope으로 소비(consume)되지 않으면 만료·삭제한다(CLAUDE_ALARM_PAIRING_PENDING_WINDOW_SECONDS, 부록 A). consumePending(nonce:within:)은 nonce 매칭(원자적)이다 — WSAuthGate actor 격리 하에서 echo된 nonce와 일치하는 항목만 점유·제거(1회 소비, replay 방지)하고 그 (tokenId, presentedSecret)를 반환하며, 동일 항목이 두 번 소비되는 일은 없다(한 번 핸드셰이크는 한 번의 첫 envelope만 증명). 일치 항목이 없거나 시간창 만료 시 삭제하고 nil. 핸드셰이크는 통과했으나 시간창 내 첫 envelope이 안 오면 연결은 인증 미승격 상태로 남아 어떤 운영 envelope도 처리되지 않는다. pendingAuth의 presentedSecret은 consume 또는 만료 즉시 메모리에서 폐기(참조 해제)하고 어떤 경로로도 로그하지 않는다.

SDK 주의 — Day 0 스파이크로 실증

setClientRequestHandler가 .reject를 반환할 때 클라이언트가 받는 관측(HTTP 상태/즉시 close)이 SDK 문서에 빈약하다. 위 코드의 게이트 ①은 Day 0 스파이크가 "reject가 클라이언트에 관측 가능"을 실증한 뒤에만 확정한다. 실증 실패 시 핸드셰이크 구조 검증 reject만 생략하는 옵션 B로 폴백한다(클로저는 전부 accept하되 헤더 Bearer를 여전히 분해해 nonce 키로 pendingAuth carry, 게이트 ②의 nonce 매칭 소비·secret 대조는 옵션 A와 동일 — carry-over 메커니즘은 유지). 어느 경우든 게이트 ②의 secret 대조가 인증의 실질이므로, 게이트 ①의 유무는 조기 차단·격리점 선확보 여부일 뿐 인증 강도에 영향이 없다. 클라이언트 진단 신호가 다르다: 옵션 A는 핸드셰이크 단계의 무신호 reject(연결 자체가 안 섬)이고, 옵션 B는 핸드셰이크 성립 후 첫 envelope에 대한 UNAUTHORIZED 에러 envelope + close다 — 클라이언트는 이 분기로 "토큰이 구조적으로 거부됐는지(A)" vs "secret이 틀렸는지(B/A 게이트②)"를 진단한다.

public struct PairingPayload: Codable, Equatable, Sendable {
    public let pairingId: String
    public let deviceTokenSecret: String   // base64url(32 random bytes). QR/claim 응답에만 존재
    public let wsEndpoint: String          // P6a: ws://127.0.0.1:<port>/ (D-5). P6b: tailnet
    public let pushChannelHint: String     // P6a: mock 채널 ID. S1/P10a 실 채널
    public let expiresAt: Date
}

public enum PairingCodec {
    // JSON → base64url → custom scheme. 라운드트립 byte-identical(Day 1 테스트).
    public static func encodeURL(_ p: PairingPayload) throws -> URL {
        let json = try JSONEncoder.iso8601().encode(p)
        let b64 = json.base64URLEncodedString()   // +/= 치환, padding 제거
        return URL(string: "claudealarm://pair?d=\(b64)")!
    }

6. ADR 운영화 — ADR-A: chatRoomId 발급 주체 + sessionId lifetime

동결 필수

master § 6 ADR 목록에서 ADR-A는 P7 freeze 전 동결 필수로 지정된다. 본 §6 본문이 그대로 docs/adr/A-chatroomid-sessionid-lifetime.md가 된다(Day 4). 모바일 트랙(P9/P10a)이 이 결정에 의존한다.

Decision (결정)

  1. chatRoomId 발급 주체 = 데스크톱. 데스크톱이 pane을 생성하는 시점에 chatRoomId를 발급하고, 이를 single source of truth로 둔다. 모바일은 발급하지 않고 데스크톱이 push/WS로 전달한 chatRoomId를 수신해 라우팅에만 쓴다.
  2. 데몬 재시작 시 모든 sessionId invalidate. 데몬이 재시작하면 in-memory SessionBindRegistry는 비어 있어 옛 sessionId는 해석 불가다. 재시작 직후 데몬은 직전 살아 있던 세션들에 대해 session.terminated(orphaned 사유)를 발신하고, chatRoomId↔sessionId 바인딩을 폐기한다. chatRoomId 자체는 pane이 영속이면 보존된다(재바인딩은 새 세션 생성 시). pane 영속(workspace JSON persistence)은 P2 스키마가 보장하며, chatRoomId의 모바일 배선 책임은 P9에 있다.

Drivers (결정 동인)

  • master § P9 "채팅룸=데스크톱 pane 1:1 바인딩" — pane은 데스크톱 소유 자원이므로 그 식별자 발급도 데스크톱이 자연 소유한다.
  • push chatRoomId 라우팅 정합 — 모바일이 발급하면 데스크톱이 모르는 ID가 push payload에 들어가 라우팅 불능. PushEnvelope.chatRoomId(현재 sessionId placeholder)의 실 source가 데스크톱이어야 일관.
  • master § P10 dedupe key 후보(messageId vs sessionId+seq)와 정합 — chatRoomId가 데스크톱 발급이면 messageId도 데스크톱 단조 발급이 가능해 dedupe 설계가 단순.

Alternatives considered (검토한 대안)

대안내용기각 사유
모바일 발급모바일이 채팅룸 진입 시 chatRoomId 생성 후 데스크톱에 통보데스크톱이 pane 소유자인데 ID를 모름 → push 라우팅 시 역매핑 테이블 필요, 두 발급원 충돌 가능. single source of truth 붕괴
합의 발급(양측 hash)pane id + device id를 해시해 양측이 같은 chatRoomId 도출pane id 변경/재생성 시 충돌, 해시 충돌 표면, 디버깅 난해. 단순 데스크톱 발급 대비 이득 없음
재시작 시 sessionId 보존데몬이 sessionId를 디스크 영속해 재시작 후 재바인딩PTY는 데몬 프로세스 자식이라 재시작 시 이미 죽음 — 보존된 sessionId가 가리킬 실 세션이 없어 거짓 생존. orphaned 신호가 정직

Why chosen (선택 이유)

데스크톱 발급 + 재시작 전부 invalidate가 "pane 소유권 = 데스크톱"이라는 master 정보 구조와 일치하고, push 라우팅·dedupe·lifecycle 매트릭스(orphaned)를 모두 모순 없이 만족하는 유일한 조합이다. 데이터 손실 우려는 메시지가 모바일 SQLite에 영속(master § P9)되므로 sessionId invalidation이 메시지를 지우지 않는다 — 세션 핸들만 끊고 채팅 이력은 보존된다.

Consequences (결과)

  • (+) push chatRoomId가 데스크톱 단조 발급이라 P10a dedupe 설계가 단순해진다.
  • (+) 데몬 재시작이 "조용한 좀비 세션" 없이 orphaned로 정직하게 노출된다.
  • (−) 모바일은 chatRoomId를 발급/예측 못 해 항상 데스크톱 통지에 의존 — 오프라인 선발급 UX 불가(MVP 범위 밖이라 수용).
  • (−) Pane.chatRoomId를 데스크톱이 채우는 코드는 P9에서 배선(P6a는 placeholder 유지) — ADR만 동결, 구현 이연.

Follow-ups (후속)

  • P9: Pane.chatRoomId 실 발급 배선 + 모바일 수신 라우팅 (master § P9).
  • P9: pane 영속 보장 책임 — 본 ADR은 "chatRoomId는 pane이 영속이면 보존"을 전제한다. pane 영속(WorkspaceManager 저장 스키마)이 chatRoomId 수명을 보장하는지 검증·배선은 P9이 책임진다(P6a는 ADR 전제만 동결, pane 저장 코드 미변경).
  • P10a: ADR-C(dedupe key) 확정 시 본 결정 참조 — messageId 데스크톱 발급 전제.
  • P7: envelope v1.0에 session.terminated orphaned 사유 코드 + pairing.claim/pairing.response 2종 kind 동결(§5.5 — P7 freeze 대상 편입).

7.1 Pre-mortem — 4 실패 시나리오

시나리오 1 — 토큰 검증 timing 누출로 토큰 탈취

증상공격자가 loopback 또는 (P6b 후) tailnet에서 Bearer 토큰을 반복 추측. 검증이 첫 불일치 바이트에서 early-return해 응답 시간이 일치 prefix 길이에 비례 → byte-by-byte 토큰 복원.
근본 원인DeviceTokenVerifier가 == 또는 String 비교(short-circuit)를 써서 timing 비의존성 위반.
완화 (계획 반영)§5.2 constantTimeEqual(누적 XOR, 길이 불일치도 상수시간). Day 1 종료 조건: grep로 short-circuit 비교 0건 측정. Day 2: 검증이 게이트 ①②에서 동일 함수 호출.

시나리오 2 — 6자리 코드 brute-force로 무단 페어링

증상공격자가 6자리(약 20비트) 코드를 claim endpoint에 대량 시도해 유효 in-flight 코드를 맞춰 디바이스 등록. 토큰을 정당하게 발급받아 이후 모든 게이트 통과.
근본 원인코드 만료(5분)만 있고 시도 횟수 제한이 없어 5분 내 10^6 시도가 가능.
완화 (계획 반영)PairingSession에 claim 시도 카운터 — 동일 코드 5회 오claim 시 즉시 폐기(rate-limit). 5분 만료 병행. Day 4 종료 조건: 5회 오claim 후 6회째 거부 + 만료 코드 거부 테스트. master § 보안 원칙과 정합.

시나리오 3 — 토큰 평문이 로그/QR 스크린샷/envelope에 노출

증상deviceTokenSecret이 os_log/디버그 print/envelope payload/UI 텍스트에 평문으로 찍혀, 로그 수집·화면 공유·패킷 캡처로 유출.
근본 원인로그·UI가 토큰 ID 대신 secret 본문을 출력. envelope이 secret을 payload에 동봉.
완화 (계획 반영)Principle 3 — secret은 QR/pairing.response/Authorization 헤더에만(운영 envelope 금지). 로그는 tokenId 앞 8자+마스킹. Day 2 종료: grep로 로그 함수에 토큰 0건. Day 3 종료: Sources/UI/에 secret 평문 0건(QR/코드만 표시). Day 0/4: 산출 문서에 토큰 0건.

시나리오 4 — claim 채널 replay / 재페어링 후 옛 토큰 잔존

증상(a) 공격자가 정당한 pairing.claim↔pairing.response 교환을 캡처해 동일 코드로 secret을 재수신(replay). (b) 디바이스를 재페어링했는데 옛 secret이 Keychain에 남아, 분실된 옛 토큰으로도 인증이 계속 성립.
근본 원인(a) claim 코드가 1회 소비되지 않고 만료까지 재사용 가능. (b) upsert가 새 secret을 추가만 하고 동일 deviceId의 옛 secret을 폐기하지 않음.
완화 (계획 반영)(a) PairingSession.claim이 성공 시 코드를 즉시 폐기(1회 소비) — 같은 코드 두 번째 claim은 PAIRING_CODE_INVALID. carry-over pendingAuth도 nonce 키로 consume 1회 — 같은 nonce echo 재전송은 소비할 항목 없어 거부(§5.3). (b) DeviceStore.upsert가 동일 deviceId 재페어링 시 옛 secret을 즉시 replace(폐기) — §5.1 주석 명문화. Day 4 종료 조건의 rate-limit 테스트에 "재페어링 후 옛 secret 검증 실패" 케이스 포함.
