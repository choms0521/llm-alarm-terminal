1. 개요

Goal

데스크톱 데몬이 push envelope을 PushTransport 추상화 뒤로 전송할 수 있게 합니다. P5에서는 실 FCM/APNs 대신 MockPushTransport를 사용하며, 모든 수락 기준은 mock 또는 단위 테스트로 측정 가능합니다. 실 FCM(Firebase 프로젝트) / 실 APNs(.p8 key) 셋업과 실 디바이스 수신은 후속(P6/S1)으로 이연합니다. 본 단계는 그 이연 항목을 측정 가능한 1급 섹션(§7)으로 기록합니다. 더불어 P4에서 이연된 부채 (g) 앱 데몬 기동과 (h) internal 입력 live 배선을 본 단계에서 함께 land 합니다.

dead-until-C4 seam: PushSender는 P5에서 production caller가 없는 deliberate-staged seam이다. 새 Claude 메시지를 감지하는 internal output tap이 C4(INTERNAL_OUTPUT_UNSUPPORTED)로 막혀 있어, 첫 production caller 배선은 C4 해소 후(§7 D7, P6+)에 일어난다. P5 테스트는 PushEnvelope을 직접 주입하여 전 경로를 검증한다.

lid-close grace supersession: 마스터 계획의 lid-close "5분 grace"는 P5에서 willSleep-즉시 무효화로 대체한다(슬립 진입 = 즉시 not-attached가 push fallback 정확성에 부합. 상세 근거는 §Day 5).

Side

desktop (macOS, Apple Silicon) + shared push envelope spec

Scope (in) — 이번 단계에서 반드시 끝낸다

  • Push envelope v0.9 모델 + codec: sessionId, messageId, preview (≤200 chars, message-boundary), chatRoomId, timestamp, fetchHint. ≤4KB validator를 codec 안에 둠.
  • PushTransport protocol seam + MockPushTransport 구현. 실 FCM/APNs transport가 깨끗이 swap-in 되는 경계를 명시.
  • PushSender: envelope 인코딩 → 4KB 검증 → transport 위임. WS-attached skip 정책 분기 포함.
  • WS-attached skip 정책: SessionBindRegistry.boundClient(forSession:) != nil을 권위 source로 질의(주입 가능 predicate seam 뒤). attached이면 skip(설정 가능 토글), 없으면 발송.
  • (g) 앱 데몬 기동: AppDelegate.applicationDidFinishLaunching이 SessionDaemon+WSServer를 기동(현재 DaemonDevCLI에만 존재). 기동 배선 로직을 testable 컴포넌트로 추출.
  • (h) internal 입력 live 배선: PaneKind.claude 세션의 live surface에 InternalSink(GhosttySurfaceInjector(surface)) 부착. surface-acquire 시점에 발화.
  • lid-close → attached 무효화: PowerEventObserver(willSleep:) 핸들러가 모든 attached binding을 not-attached로 무효화 → push fallback 경로 발동. 무효화 로직을 testable 함수로 추출.
  • push skip 토글 최소 설정 UI(진입점만).
  • 환경 변수: CHAT_TERMINAL_FCM_KEY_ID, CHAT_TERMINAL_APNS_KEY_ID(필수, Keychain item ID). 에러 코드: PUSH_PAYLOAD_TOO_LARGE.

Out of scope — 이번 단계에서 절대 손대지 않는다

  • 실 FCM HTTP v1 transport, 실 APNs HTTP/2 transport (P6/S1). ADR-D는 deferred-decision stub으로만 남김.
  • 실 디바이스/시뮬레이터 push 수신, 실 Firebase 콘솔/.apns drag-drop 검증 (P6/S1).
  • pairing flow, device registry, WS auth, 토큰 lifecycle (P6a/P6b).
  • 모바일 측 push 수신 / fetchHint resolver API (P6/S1).
  • chatRoomId 실 매핑 source (아직 없음 → sessionId placeholder로 reserved).
  • push trigger source — 새 Claude 메시지를 감지하여 PushSender.send()를 호출하는 주체. P4가 internal output 스트림을 미지원으로 막았으므로(SessionDaemon.requestInternalOutput → INTERNAL_OUTPUT_UNSUPPORTED, C4), "무엇이 새 메시지를 감지하는가"는 P6+ 설계 질문이다. 2. RALPLAN-DR Summary

Principles (5)

  1. Mock-first, infra-deferred. 외부 인프라(FCM/APNs)는 P5의 측정 대상이 아니다. 모든 수락 기준은 MockPushTransport 또는 단위 테스트로 측정 가능해야 한다. 실 transport는 PushTransport protocol 뒤로 swap-in 되며, 그 경계가 본 단계의 1급 산출물이다.
  2. Reject, not drop. 4KB 초과 payload는 조용히 잘리거나 버려지지 않고 명시적으로 거부되며 PUSH_PAYLOAD_TOO_LARGE가 로깅된다. P4의 ring-buffer drop-mark 정책과 구별되는, push 경로 고유의 reject 정책이다.
  3. Korean-safe message-boundary preview. preview는 ≤200 chars이되 UTF-8 다바이트(한국어) 경계에서만 자른다. P4 drop-and-mark 원칙(byte 경계 손상 금지)을 push preview에도 적용한다.
  4. Testable seam over AppKit coupling. (g)/(h)/lid-close 부채는 AppKit lifecycle에 닿지만, 의사결정 로직은 DaemonTests에서 device/app-launch/network 없이 실행 가능한 seam으로 추출한다. 얇은 AppKit 배선은 비-게이팅 매뉴얼 노트로 남긴다.
  5. Authoritative attached source. "WS attached" 판정은 단일 권위 source(SessionBindRegistry.boundClient(forSession:) != nil)에서만 나온다. push-skip 정책과 lid-close 무효화가 동일 source를 질의·변경한다.

Decision Drivers (Top 3)

  1. 인프라 없이 측정 가능해야 한다. Firebase 프로젝트와 APNs .p8 key가 아직 없으므로, P5 exit가 실발송에 의존하면 측정 불가가 된다. 모든 결정은 "mock으로 측정 가능한가"로 평가한다.
  2. (g)/(h) 부채를 P5에서 닫아야 한다. P4가 앱 데몬 기동과 internal 입력 live 배선을 이연했고, push fallback 경로(lid-close → attached 무효화)는 이 두 배선이 land해야 end-to-end로 검증된다. 부채 closure가 push 정책의 전제 조건이다.
  3. 실 transport swap-in 비용 최소화. P6/S1에서 실 FCM/APNs를 붙일 때 PushSender/codec/skip 정책을 다시 짜지 않도록, 변경 지점을 transport 구현 1개로 격리한다.

Day 1 — Push envelope v0.9 codec + 4KB validator + 한국어 message-boundary preview

목표
v0.9 envelope을 직렬화/역직렬화하고, ≤4KB validator를 codec 안에 두며, 한국어 포함 ≤200 chars message-boundary preview를 구현합니다.
산출물
  • Sources/Push/PushEnvelopeCodec.swift — encode/decode + 4KB validator(reject + PUSH_PAYLOAD_TOO_LARGE)
  • Sources/Push/PreviewBuilder.swift — ≤200 chars, UTF-8/grapheme 경계 보존 절단
  • Tests/DaemonTests/PushEnvelopeCodecTests.swift, PreviewBuilderTests.swift
OMC 위임
executor (sonnet) → tdd-guide 보조 (RED→GREEN)
예상 소요
1.0일
기술 노트
  • 4KB 한도는 인코딩된 최종 payload byte 길이 기준으로 검증한다. 다른 필드(UUID, timestamp, chatRoomId=sessionId, ≤200 char preview)는 모두 bounded이므로, 4KB를 실제로 넘길 수 있는 유일한 필드는 unbounded fetchHint다. 따라서 validator는 codec 안(인코딩 직후 최종 byte 길이)에 두고, A1 테스트는 큰 fetchHint로 4097-byte payload를 구성한다.
  • preview 절단은 Swift String.prefix(200)을 사용한다. Swift String.prefix(n)은 Character(extended grapheme cluster) 단위로 동작하므로 한국어 조합 문자(NFD 분해된 자모 포함)가 경계에서 깨지지 않는다. UnicodeScalar 루프를 직접 작성할 필요가 없다(P4 message-boundary 원칙과 동일하게 grapheme 경계 보존).
  • chatRoomId는 reserved: 인코딩 시 sessionId와 동일 값 placeholder를 채운다(매핑 source 부재).
  • shared push envelope spec이므로 JSONEncoder 전략을 P5에서 고정한다: dateEncodingStrategy = .millisecondsSince1970(epoch-millis), outputFormatting = [.sortedKeys, .withoutEscapingSlashes]. P6 모바일 파서가 swap-in 없이 동일 형태를 읽도록 P5에서 박아 P6 비용을 절감한다.

종료 조건

  • xcodebuild test -scheme DaemonTests -only-testing:DaemonTests/PushEnvelopeCodecTests exit code 0
  • 4097-byte payload(큰 fetchHint로 구성) 인코딩 시 codec이 PushError.payloadTooLarge를 throw하고 XCTAssertEqual(error.code, "PUSH_PAYLOAD_TOO_LARGE")가 통과한다(drop 아닌 reject)

5.1 PushEnvelope v0.9 + codec (4KB validator 내장)

public struct PushEnvelope: Codable, Equatable, Sendable {
    public let sessionId: UUID
    public let messageId: UUID
    public let preview: String        // ≤200 chars, message-boundary
    public let chatRoomId: String     // reserved: P5에서 sessionId placeholder
    public let timestamp: Date
    public let fetchHint: String?
}

public enum PushError: Error {
    case payloadTooLarge   // code = "PUSH_PAYLOAD_TOO_LARGE"
    var code: String { switch self { case .payloadTooLarge: return "PUSH_PAYLOAD_TOO_LARGE" } }
}

public enum PushEnvelopeCodec {
    static let maxPayloadBytes = 4096   // 4KB push 한도

    // shared push envelope spec: 인코더 전략을 P5 에서 고정하여 P6 모바일 파싱이
    // swap-in 없이 동작하게 한다. timestamp 는 epoch-millis(Int64), 키 순서 고정.
    static let encoder: JSONEncoder = {
        let e = JSONEncoder()
        e.dateEncodingStrategy = .millisecondsSince1970
        e.outputFormatting = [.sortedKeys, .withoutEscapingSlashes]
        return e
    }()
    static let decoder: JSONDecoder = {
        let d = JSONDecoder()
        d.dateDecodingStrategy = .millisecondsSince1970
        return d
    }()

    // 4KB 검증을 codec 안(인코딩 직후)에 둔다 — preview 절단 후에도 unbounded
    // fetchHint 가 한도를 넘길 수 있으므로 최종 byte 길이로 reject.
    public static func encode(_ env: PushEnvelope) throws -> Data {
        let data = try encoder.encode(env)
        guard data.count <= maxPayloadBytes else {
            throw PushError.payloadTooLarge   // reject, drop 아님
        }
        return data
    }
}

5.2 PreviewBuilder — 한국어 grapheme 경계 보존

public enum PreviewBuilder {
    static let maxChars = 200

    // grapheme cluster(Character) 단위 절단으로 한국어 조합 문자가

    public static func build(from message: String) -> String {

    func isAttached(_ sessionId: UUID) async -> Bool {
        await registry.boundClient(forSession: sessionId) != nil
    }

public actor PushSender {
    let transport: any PushTransport
    let attachment: any AttachmentQuerying
    let config: PushPolicyConfig   // skipWhenAttached: Bool = true

    public func send(_ env: PushEnvelope, target: PushTarget) async {
        // 1) WS-attached skip 분기 (주입된 predicate seam)
        if config.skipWhenAttached, await attachment.isAttached(env.sessionId) {
            return   // foreground 사용자 → WS 스트림이 처리, push skip
        }
        // 2) encode + 4KB reject (codec 내부 validator)
        let data: Data
        do { data = try PushEnvelopeCodec.encode(env) }
        catch { PushLog.error("PUSH_PAYLOAD_TOO_LARGE"); return }  // reject, transport 미호출
        // 3) transport 위임 (P5: mock, P6/S1: 실 FCM/APNs swap-in)
        await transport.send(data, target: target, sessionId: env.sessionId)
    }
}

6. ADR 운영화 (ADR-D deferred-decision stub 포함)

6.1 본 단계 ADR 적용

  • §4.5 보안 정책 적용: FCM/APNs key는 Keychain 전용. 코드/로그에 직접 노출 금지. PushSender/transport는 Keychain item ID(CHAT_TERMINAL_FCM_KEY_ID, CHAT_TERMINAL_APNS_KEY_ID)만 참조하며, key material을 메모리에 장기 보관하지 않는다. PushLog는 payload preview를 로깅하되 key/token은 절대 로깅하지 않는다.
  • Authoritative attached source: skip 정책과 lid-close 무효화가 동일하게 SessionBindRegistry를 권위 source로 삼는다(이중 판정 source 금지).
  • Push/WS 프로토콜 분리: push는 PushEnvelope로 분리하여 WS 프로토콜(EnvelopeKind)을 오염시키지 않는다(Option C 거부 근거를 정책화).

6.2 ADR-D — APNs HTTP/2 라이브러리 선택 (deferred-decision stub)

ADR-D 상태: DEFERRED (P5에서 확정하지 않음)

결정 대상: 실 APNs HTTP/2 transport를 Swift native(Network.framework/URLSession HTTP/2)로 구현할 것인가, 외부 패키지를 도입할 것인가.

왜 deferred인가: P5는 mock transport만 구현하므로(MockPushTransport) 실 라이브러리 선택은 P5 exit의 전제가 아니다. 지금 라이브러리를 강제 선택하면 실 통합 시점(P6/S1)의 제약 조건(인증 방식, 토큰 갱신, connection pooling)을 알기 전 결정이 되어 재논의 비용이 든다.

최종 선택 기준(P6/S1에서 평가):

  • APNs token-based auth(.p8 JWT) 갱신 주기(≤1h)를 native로 다룰 수 있는가
  • HTTP/2 multiplexing + connection 재사용을 표준 라이브러리로 충분히 다루는가
  • 외부 패키지 도입 시 코드사이닝/노타라이즈 영향(library validation entitlement)
  • FCM HTTP v1과 transport 추상화를 공유 가능한가

금지: P5에서 특정 라이브러리를 강제 선택하지 않는다. 마커만 남기고 기준만 나열한다.
