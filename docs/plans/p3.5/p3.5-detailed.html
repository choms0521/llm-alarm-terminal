1. 개요

Goal

2. RALPLAN-DR Summary

Principles (5)

  1. 영속 schema 무결성 우선. workspaces.json은 사용자의 작업 상태가 그대로 영속화된 파일이므로 v1 → v2 migration은 backward-compat 자동 변환 + .bak 보관을 의무로 한다. hard cut으로 빈 상태 시작은 금지(P2 lessons learned).
  2. TUI 영감 자체 구현. agent-view는 Anthropic 공식 claude agents TUI의 카드 그리드 패러다임을 SwiftUI로 옮긴 것이었으나, 결정자 원의도는 TUI 영감을 받은 자체 2-pane 통합 콘솔. OutlineGroup 3단 트리 + GhosttyTerminalView로 SwiftUI 네이티브 구현하되, status 시각화 디테일은 TUI에서 차용.
  3. P3 인프라 최대 재사용. SessionStatusObserver/NeedsInputPolicy/AnsiSGRParser/ShellPreviewExtractor/Utf8BoundaryTruncator/FocusedPaneStore/ViewportPollingTimer/GhosttyViewportProvider는 데이터 인프라이며 UI와 독립적이므로 그대로 보존. 폐기는 AgentDashboardView/AgentCardView/AgentSortFilterControls/AgentViewSettings/AgentJumpAction UI 5건만. P3 6일 작업 중 데이터 인프라 60%는 보존, UI 40%만 폐기.
  4. 검증 부실 재발 방지. P3 시연에서 D-1~D-4가 드러난 원인은 manual UX walkthrough 부재. P3.5에서는 Day 1과 Day 8에 각각 walkthrough 체크리스트를 명시적으로 통과시키고, master plan v5 진화 시 phase 종료 의무 조항으로 박는다.
  5. 사용자 자료 손실 zero. P3.5의 본질은 (i) workspaces.json schema 진화에서 사용자 작업 상태 손실 금지, (ii) CLAUDE_CONFIG_DIR 격리 폐지 시 사용자 ~/.claude credentials/projects 보존, (iii) P3 카드 그리드 폐기 시 누적된 사용자 sort/filter 선호도("agentView.settings") cleanup 시 1회 OSLog 기록 의무. 세 영역 모두 사용자 자료 보호가 본 wave의 무조건 invariant이며, 본 principle은 R1/R3/R6 risk 전체를 회수한다. // Sources/Workspace/WorkspaceSchemaV1.swift (신규, read-only)

// Sources/Workspace/WorkspaceSchemaMigration.swift (신규)
public enum WorkspaceSchemaMigration {
    /// 부팅 시 1회 실행. v1 감지 시 v2 로 변환하고 v1 원본은 별도 backup 파일로 보존.
    /// backup 파일명을 .bak 와 분리하는 이유: P2 atomic write 가 .bak 를 직전 정상본 의미로
    /// 이미 사용 중. migration backup 은 영속 보존이 목적이므로 의미가 다름.
    public static func migrateIfNeeded(at url: URL) throws -> Bool {
        let data = try Data(contentsOf: url)
        guard isV1(data: data) else { return false }  // version >= 2 면 no-op
        let v1 = try decodeV1(data: data)
        let v2 = convert(v1: v1)
        // 1. v1 원본 backup (실패 시 abort, migration 미수행 → 사용자 다이얼로그 표시)
        let ts = ISO8601DateFormatter().string(from: Date())
        let backupURL = url.deletingLastPathComponent()
            .appendingPathComponent("workspaces.json.v1-pre-migration-\(ts)")
        try FileManager.default.copyItem(at: url, to: backupURL)
        // 2. v2 overwrite (WorkspaceFile.version = 2 stamp)
        try encodeV2(v2: v2, to: url)
        return true  // 사용자에게 성공 알림 (한국어 다이얼로그)
    }

    private static func convert(v1: [V1Workspace]) -> [Workspace] {
        v1.map { v1Ws in
            let v2Panes = v1Ws.panes.map { v1Pane in
                let position: PanePosition = (v1Pane.position == "top") ? .left : .right
                // v1 의 단일 sessionId → 단일 Tab 으로 wrap
                let tab = Tab(
                    id: UUID(),
                    sessionId: v1Pane.sessionId,
                    kind: v1Pane.kind,
                    name: defaultName(for: v1Pane.kind),
                    createdAt: Date()
                )
                return Pane(
                    id: v1Pane.id,
                    position: position,
                    tabs: [tab],
                    activeTabId: tab.id,
                    chatRoomId: v1Pane.chatRoomId,
                    extraFields: v1Pane.extraFields
                )
            }
            return Workspace(... panes: v2Panes, ...)
        }
    }
}
migration 안전 invariant

v1 → v2 변환은 (i) 백업 파일 workspaces.json.v1-pre-migration-<ISO8601> 생성이 먼저 성공해야 v2 overwrite 진행 (P2 atomic write의 .bak은 직전 정상본 의미이므로 별도 명명으로 충돌 회피), (ii) backup copy 실패 시 abort + 사용자에게 한국어 다이얼로그 "워크스페이스 스키마를 v2로 변환하려 했으나 백업 생성에 실패했습니다. 원본 파일을 보존합니다." 표시 후 v1 파일 그대로 보존, (iii) migration 성공 시 사용자에게 "워크스페이스 스키마를 v2로 안전하게 변환했습니다. 이전 버전은 workspaces.json.v1-pre-migration-...에 보관됩니다." 알림, (iv) overwrite된 파일은 WorkspaceFile.version = 2로 stamp되어 두 번째 부팅 시 (version ?? 1) < 2 검사가 false 반환 → migration no-op (idempotent).

// Sources/Workspace/WorkspaceManager.swift 확장
extension WorkspaceManager {

    @MainActor
    public func closeTab(workspaceId: UUID, paneId: UUID, tabId: UUID) async throws {
        // 1. 해당 tab 의 session terminate (lifecycle hook 발동)
        if let sessionId = findTab(workspaceId: workspaceId, paneId: paneId, tabId: tabId)?.sessionId {
            try await sessionManager.terminate(id: sessionId)
        }
        // 2. tab 제거 + activeTabId 갱신
        removeTab(workspaceId: workspaceId, paneId: paneId, tabId: tabId)
        // 3. cascade: pane.tabs.isEmpty → pane 제거
        if let pane = findPane(workspaceId: workspaceId, paneId: paneId), pane.tabs.isEmpty {
            removePane(workspaceId: workspaceId, paneId: paneId)
        }
        // 4. cascade: workspace.panes.isEmpty && workspace.canClose → workspace 제거
        if let ws = findWorkspace(id: workspaceId), ws.panes.isEmpty, ws.canClose {
            removeWorkspace(id: workspaceId)
        }
    }
}
  • cascade 순서: tab → pane → workspace. 각 단계는 invariant 점검 후 다음 단계 진입
  • workspace.canClose == false (agent-view) 인 경우 마지막 pane이 비어도 workspace는 제거되지 않음 (자연 보호, 추가 분기 불필요)
  • agent-view 자체는 pane/tab 구조를 가지지 않으므로 본 정책에서 제외 (invariant: agent-view는 영구)
  • 단축키 재매핑: Cmd+W = 활성 탭 close (cascade 발동), Cmd+Shift+W = workspace close (직접 호출, agent-view는 메뉴에서도 비활성)

6.1 WorkspaceManager.closeTab cascade (핵심)

@MainActor
public func closeTab(workspaceId: UUID, paneId: UUID, tabId: UUID) async throws {
    // Phase 1: tab의 session terminate (lifecycle hook 발동)
    guard let ws = workspaces.first(where: { $0.id == workspaceId }),
          let paneIdx = ws.panes.firstIndex(where: { $0.id == paneId }),
          let tabIdx = ws.panes[paneIdx].tabs.firstIndex(where: { $0.id == tabId })
    else { throw WorkspaceError.tabNotFound }

    let closingTab = ws.panes[paneIdx].tabs[tabIdx]
    if let sessionId = closingTab.sessionId {
        try await sessionManager.terminate(id: sessionId)
        surfaceRegistry.detach(tabId: tabId)
    }

    // Phase 2: tab 제거 + activeTabId 갱신 (immutable rebuild)
    var newTabs = ws.panes[paneIdx].tabs
    newTabs.remove(at: tabIdx)
    let newActiveTabId: UUID? = {
        if ws.panes[paneIdx].activeTabId == tabId {
            return newTabs.first?.id  // sibling 으로 이동
        } else {
            return ws.panes[paneIdx].activeTabId  // 그대로
        }
    }()
    let newPane = ws.panes[paneIdx].with(tabs: newTabs, activeTabId: newActiveTabId)

    // Phase 3: cascade — pane.tabs.isEmpty → pane 제거
    var newPanes = ws.panes
    if newPane.tabs.isEmpty {
        newPanes.remove(at: paneIdx)
    } else {
        newPanes[paneIdx] = newPane
    }

    // Phase 4: cascade — workspace.panes.isEmpty && canClose → workspace 제거
    if newPanes.isEmpty && ws.canClose {
        removeWorkspace(id: workspaceId)
        // agent-view 는 canClose=false 이므로 이 분기 진입 불가
    } else {
        replace(workspace: ws.with(panes: newPanes))
    }

    try store.save(workspaces)  // 영속화
}

6.2 AgentTreeSelection.selectFirstAvailable (REQ-5 default)

@MainActor
public final class AgentTreeSelection: ObservableObject {
    @Published public var selectedTabId: UUID? = nil
    @Published public var expandedNodeIds: Set<UUID> = []  // workspace.id

    /// 트리 DFS 순회 첫 sessionId 보유 tab. 모든 tab이 sessionId nil 이면 첫 tab.
    /// 트리에 tab 자체가 없으면 selectedTabId = nil.
    public func selectFirstAvailable(manager: WorkspaceManager) {
        for ws in manager.workspaces where ws.kind == .normal {
            expandedNodeIds.insert(ws.id)  // default expand
            for pane in ws.panes {
                for tab in pane.tabs where tab.sessionId != nil {
                    selectedTabId = tab.id
                    return
                }
            }
        }
        // fallback: 첫 tab (sessionId nil 이어도)
        selectedTabId = manager.workspaces
            .first(where: { $0.kind == .normal })?
            .panes.first?.tabs.first?.id
    }
}

6.3 WorkspaceSchemaMigration v1 감지 + 변환

internal func isV1(data: Data) -> Bool {
    // 휴리스틱: WorkspaceFile.version 필드 기준 + pane shape 보강.
    // P2 v1 은 이미 `version: 1` 로 stamp 되어 있으므로 version 필드를 1차 판정자로,
    // pane shape (position in {top,bottom} 또는 tabs 부재+sessionId 존재) 을 2차 보강으로 사용.
    guard let json = try? JSONSerialization.jsonObject(with: data) as? [String: Any]
    else { return false }

    // 1차: version field 검사 (v2 이상이면 즉시 false → idempotent)
    let version = (json["version"] as? Int) ?? 1  // 부재 시 v1 가정
    guard version < 2 else { return false }

    // 2차: pane shape 보강 — version 만으로 잘못 stamp 된 케이스 방어
    guard let wsArray = json["workspaces"] as? [[String: Any]],
          let firstWs = wsArray.first,
          let panes = firstWs["panes"] as? [[String: Any]]
    else { return true }  // version=1 이고 빈 workspaces 면 v1 으로 처리 (stamp 만 갱신)

    return panes.contains { ($0["position"] as? String) == "top" ||
                            ($0["position"] as? String) == "bottom" } ||
           panes.contains { $0["tabs"] == nil && $0["sessionId"] != nil }
}

7.2 ADR-F (신설) — Pane을 멀티탭 컨테이너로

Decision: Pane은 단일 session을 갖던 단일-탭 모델에서 멀티탭 컨테이너로 변경. Tab { id, sessionId, kind, name, createdAt, extraFields } 도입.

Drivers: (D1) 일상 사용 baseline — IDE/Ghostty/iTerm2 표준 형태, (D2) P4 protocol freeze 전 모델 안정화, (D3) SurfaceRegistry key 단위를 명확히.

Alternatives: (A) workspace 단위 탭바 — REQ-2 의도와 어긋남, 거부 / (B) pane 단위 탭바 + Tab 모델 — 선택 / (C) Pane.sessions[] 배열만 — Tab 메타데이터 부재, 거부.

Consequences: 긍정 — sessionId 1:1 wrap으로 v1 migration 자연 / 부정 — SurfaceRegistry key 변경 (paneId → tabId) / Follow-ups: P4에서 protocol freeze 시 Tab.id를 protocol 단위로 노출할지 결정.

7.3 ADR-G (신설) — CLAUDE_CONFIG_DIR 격리 정책

Decision: CLAUDE_CONFIG_DIR 강제 격리 폐지. 사용자 ~/.claude 공유.

Drivers: (D1) 일상 사용 불가의 직접 원인 (매번 재로그인), (D2) Anthropic 공식 문서가 ~/.claude 공유를 default로 명시, (D3) P1/P2의 세션별 credential 격리 가설은 검증 부재.

Alternatives: (A) 격리 폐지 — 선택 / (B) 사용자 opt-in 격리 (env 변수로 toggle) — P3.5 범위 초과, 거부 / (C) 부분 격리 (claude.json만 격리하고 credentials은 공유) — Anthropic CLI 내부 동작 불확실, 거부.

P4로의 인계 사항

P4 (WebSocket 채널 / claude session 자체의 영속화·resume)는 P3.5에서 v2로 안정화된 schema를 base로 진행한다. WS envelope의 session bind 단위는 sessionId (v1과 동일)이며, tab 메타데이터(name, createdAt 등)는 protocol에 노출하지 않는다. P3.5에서 보존된 status 인프라는 P4 WS envelope의 status field source로 그대로 재사용 가능.
