## 2. v1 → v2 surgical edits (Architect 권고 반영)

Architect가 v1 검토에서 발행한 NEEDS_EDIT 5건 + NEEDS_DEEPER_CHANGE 1건을 Planner v2에서 모두 흡수.

### Edit 1 (CRITICAL): schema migration `isV1` 휴리스틱 정정

**Architect 지적**: v1 plan은 `json["schemaVersion"] as? Int != 2`를 v1 감지 휴리스틱으로 사용. 그러나 실제 코드(`Sources/Workspace/WorkspaceStore.swift:5-7`)는 필드명이 `version`이며, P2 v1은 이미 `version: 1`로 stamp되어 있음. 또한 `.bak` 백업 명명은 P2 atomic write의 직전 정상본 보관용과 충돌.

**v2 정정**:
- `isV1(data:)` 휴리스틱: `(file.version ?? 1) < 2` AND (`pane.position in {"top","bottom"}` OR `pane.sessionId 필드 존재 && pane.tabs 부재`)
- migration 백업 명명: `workspaces.json.v1-pre-migration-<ISO8601>` (atomic write의 `.bak`와 별개)
- 한국어 에러 다이얼로그 카탈로그 2건 신규 추가:
  - "워크스페이스 스키마를 v2로 변환하려 했으나 백업 생성에 실패했습니다. 원본 파일을 보존합니다."
  - "워크스페이스 스키마를 v2로 안전하게 변환했습니다. 이전 버전은 `workspaces.json.v1-pre-migration-...` 에 보관됩니다."
- Day 2 종료 조건 grep target 정정: `schemaVersion` → `version`

### Edit 2 (CRITICAL): Day 0 진입 게이트 분리 + 동시 claude tab walkthrough 추가

**Architect 지적**: v1 plan의 Day 1이 "R-1 회귀 검증(반나절) + REQ-3 CLAUDE_CONFIG_DIR 폐지(반나절)"로 묶여 있어, R-1 회귀 발견 시 Day 1이 막힘. 또한 REQ-2(멀티탭) 도입으로 한 pane에 claude tab 2개 동시 spawn 가능 → 한 workspace에 claude 최대 4개 → `~/.claude/claude.json` file lock 충돌 가능성이 R6 risk로 식별됐으나 walkthrough가 누락.

**v2 정정**:
- **Day 0 (P3.5 진입 게이트, 0.5일 신설)**: R-1 회귀 검증만 단독 수행 (Backspace / Enter / Tab / Arrow / 한글 IME / 사이드바 표시 / 첫 시작 agent-view 자동 선택)
- **Day 1 (1일)**: REQ-3 격리 폐지 + manual UX walkthrough 카탈로그 정착 + "2개 claude tab 동시 spawn → 5분 idle → 양쪽 모두 정상 동작" walkthrough 추가
- Day 8 acceptance walkthrough A3에 동시 claude session 명시
- Effort 표기: Day 0는 게이트이므로 phase budget 8 Day 외 +0.5일

### Edit 3 (HIGH): 좌측 트리 vs 사이드바 navigation 의미 분기 + Risk R8 신설

**Architect 지적**: v1 plan은 agent-view 좌측 트리(workspace-pane-tab 3단)가 사이드바(workspace 목록)와 정보 중복인 점을 risk로 식별하지 않음. 두 navigation 체계의 의미 분기 규칙도 명세 없음.

**v2 정정**:
- D3-6 본문에 navigation 의미 분기 규칙 명시:
  - 사이드바 클릭 → workspace 선택 변경 + 좌측 트리에서 해당 ws 자동 expand
  - 좌측 트리 클릭 → tab 선택 변경 + 사이드바 selectedID는 클릭한 tab의 부모 workspace와 어긋날 때만 동기화
- Risk R8 (정보 아키텍처 중복) 신설: 가능성 MED / 영향 MED / 완화 = 동기화 규칙 + Day 4 walkthrough에서 사용자 혼란 평가
- 결정자 청취 Q-α-1 신설: "agent-view 활성 시 사이드바 시각 처리 — 240px 유지 / 180px 축소 / collapse 가능 옵션" (default 240px 유지)

### Edit 4 (MED): 컨틴전시 +1일 → +2일 격상

**Architect 지적**: v1 plan의 컨틴전시 1일은 R1(schema migration)만 cover. R4(surface lifecycle) / R5(polling overhead) / R8(정보 중복) risk가 빈 줄.

**v2 정정**:
- 총 estimate: 8 Day + Day 0 게이트 0.5일 + 컨틴전시 2일 = 최대 10.5일
- 컨틴전시 분배: schema migration +0.5일 / surface lifecycle +0.5일 / UI redesign +0.5일 / 동시 claude session +0.5일

### Edit 5 (MED): Principle 5 격하 → 교체

**Architect 지적**: v1 plan의 Principle 5 "Day별 commit 안정성"은 진짜 principle이 아니라 operational pattern (means). 진짜 principle은 그 뒤에 있어야 함.

**v2 정정**:
- Principle 5 교체: **"사용자 자료 손실 zero"**
- 근거: schema migration / CLAUDE_CONFIG_DIR 격리 폐지 / cleanup 스크립트 / 폐기 파일 모두 사용자 워크스페이스 + claude credentials 보존이 본질. R1/R3/R6 모두 본 principle로 회수.
- Day별 commit pattern은 operational pattern으로 § 4 day-by-day 본문에 적시(격하 보존)

### Edit 6 (LOW): 중간 walkthrough Day 3/Day 5 보강

**Architect 지적**: v1 plan은 Day 1과 Day 8 두 walkthrough만 명시. Day 4-5의 큰 UI 변경(agent-view 재설계) 시점에 중간 검증이 없어 Principle 4(검증 부실 재발 방지) 부분 위반.

**v2 정정**:
- Day 3 종료 조건에 "schema v2 migration 후 실제 v1 fixture로 자동 변환 walkthrough 1회" 추가
- Day 5 종료 조건에 "agent-view 2-pane split 진입 + 좌측 트리 선택 + 우측 surface 갱신 walkthrough 1회" 추가
- A10 acceptance를 "5회 walkthrough Day 0/1/3/5/8 의무" 명시로 강화

### v1 → v2 변경 통계

> 본문 lines: 1,629 → 1,723 (+94 lines)  
> 신규 섹션: Day 0 게이트 카드 / Risk R8 / Q-α-1 결정자 청취 / 한국어 에러 카탈로그 2건  
> 정정 site: Principle 5 / Effort 표 / Day 2 종료 조건 grep target / 의사 코드 / A7 / A10 / R1 mitigation
