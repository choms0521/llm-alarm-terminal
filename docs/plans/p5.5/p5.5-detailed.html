1. 개요

Goal

agent-view를 카드 그리드에서 좌우 스플릿 작업 화면으로 진화시킨다. 좌측은 workspace → pane → tab(세션) 3단 트리, 우측은 클릭한 세션의 기존 라이브 터미널(같은 PTY, 같은 scrollback)을 즉시 호스팅한다.

Side

desktop (macOS, Apple Silicon)

Scope (in) — 이번 단계에서 반드시 끝낸다

  • 좌측 OutlineGroup 3단 트리(workspace/pane/tab) — WorkspaceManager.workspaces + SessionStatusCoordinator.snapshots 재사용, 신규 영속 모델 0건
  • 우측 라이브 터미널 호스트 — SurfaceRegistry.acquireExisting(id: tabId)로 mount하는 NSViewRepresentable + 재부모화 규율
  • 선행 리팩터링: PaneTerminalView.buildCommand를 공용 타입(TerminalCommandBuilder)으로 추출 — agent-view 우측이 lazy 미생성 탭 클릭 시 신규 spawn할 때 동일 로직 사용
  • 트리 row의 status 뱃지 — 기존 AgentStatusBadge 재활용
  • 사이드바 ↔ 좌측 트리 navigation 의미 분기 규칙(P3.5 D3-6 R8 규칙 계승)
  • 기존 카드 그리드 자산 처리(폐기) — § 2.3 옵션 C 결정 적용
  • 알려진 제약 3건의 설계·리스크·수락 기준 반영

Scope (out) — 이번 단계에서 다루지 않는다

  • 우측 다중 분할(동시 2개 이상 세션 표시) — 각 세션이 독립 surface라 기술적으로 가능하나 MVP에서는 단일 우측 호스트만. post-P5.5 토글
  • 트리 drag-and-drop 재배치, 트리에서의 workspace/pane/tab 생성·삭제(생성·삭제는 기존 normal workspace UI 책임)
  • 좌측 트리 정렬 토글(최근 활동순) — P3.5 D3-6 Q-β대로 workspace 추가순 default 고정, 토글은 post-MVP
  • 우측 surface fade transition(P3.5 Q-γ대로 즉시 교체)
  • W-3-1 수동 sign-off(독립 부채로 잔존)

2. RALPLAN-DR Summary

2.1 Principles (설계 원칙)

  1. 신규 영속 모델 0건 — 트리는 WorkspaceManager.workspaces(이미 영속) + SessionStatusCoordinator.snapshots(런타임)의 파생 view일 뿐이다. 새 저장 스키마/마이그레이션을 만들지 않는다.
  2. 세션 공유는 surface 재부모화로만 — 우측에 뜨는 터미널은 항상 acquireExisting이 반환한 기존 NSView다. 절대 같은 tabId에 대해 두 번째 GhosttyTerminalView를 생성하지 않는다(ADR-I 1 surface = 1 NSView 불변식).
  3. 순수 함수와 UI 분리 — 트리 빌드(AgentTreeBuilder)와 선택 로직(selectFirstAvailable)은 SwiftUI 비의존 순수 함수로 두어 단위 테스트로 측정한다. libghostty 의존은 AgentTerminalHostView 한 곳에 격리한다.
  4. 기존 테스트 비파괴 우선 — 사전 집계 330건의 테스트 함수가 green인 상태를 매 Day 종료 시 회귀 검증한다. 카드 그리드 자산을 건드릴 때 관련 테스트의 운명(이전/삭제)을 명시적으로 처리한다.
  5. 알려진 제약을 숨기지 않고 UI/문서에 노출 — 동시 미러링 불가·lazy spawn·리사이즈 공유는 버그가 아니라 ADR-I의 본질적 결과다. 수락 기준과 리스크 표에서 측정 가능한 형태로 확인한다.

2.2 Decision Drivers (Top 3)

  1. 세션 공유의 무결성 — 우측 클릭 시 새 셸이 아니라 같은 PTY/scrollback이 떠야 한다. 이것이 본 단계의 존재 이유다. 재부모화는 코드베이스에 선례 없는 미검증 가정이므로, Day 2 GUI 스파이크가 실패하면 read-only 미러 fallback으로 전환한다. 스파이크 게이트가 단계 성패의 1차 분기다.
  2. 회귀 비용 최소화 — 데몬/모바일 트랙과 독립한 데스크톱 UI 변경이므로, 백엔드 69개 핵심 테스트와 워크스페이스 schema를 절대 건드리지 않는 격리가 가능하고 또 필수다.
  3. P3.5 설계 자산 재활용 vs 신선한 결정 — P3.5 D3-6/D3-7이 이미 트리 컴포넌트 청사진과 카드 그리드 폐기 매핑을 남겼다. 이를 그대로 따를지, master-plan-v2가 연 "보존 또는 흡수" 선택지를 재평가할지가 핵심 분기다.

2.3 Viable Options — 카드 그리드 보존 vs 트리 흡수

master-plan-v2 § 3은 "기존 카드 그리드/점프는 보존 또는 트리로 흡수(상세 계획에서 결정)"라 명시적으로 본 계획서에 결정을 위임했다. 3개 옵션을 비교한다.

옵션내용장점단점
A. 완전 흡수(카드 그리드 폐기) P3.5 D3-7 매핑대로 AgentDashboardView/AgentCardView/AgentDashboardViewModel/AgentSortFilterControls/AgentViewSettings/AgentJumpAction 폐기. 트리가 유일한 agent-view UI. 단일 정보 아키텍처(혼란 없음). 코드 표면 축소. P3.5 결정과 정합. dead 자산 미잔존. 관련 테스트 ~6파일 삭제/이전 필요. 카드 그리드 선호 사용자에 회귀(단, 미출시 기능이라 실사용자 없음). extraFields["agentView.settings"] cleanup 필요.
B. 병존(탭 토글) agent-view 상단에 "그리드/스플릿" 세그먼트 토글. 두 UI 공존. 점진 이행. 카드 그리드 자산 보존. 정보 아키텍처 이중화. 두 코드 경로 유지 비용. 카드의 AgentJumpAction(워크스페이스 점프)과 트리 클릭(우측 호스트)의 의미 충돌. master-plan-v2의 "진화" 의도와 어긋남.
C. 트리 우선 + 그리드 데이터 레이어만 보존 UI는 트리로 교체하되 순수 데이터 인프라(observer/coordinator/badge/SessionIndex)는 P3.5 D3-7 "보존" 매핑대로 유지. 트리 빌더는 동일 데이터 인프라 위에서 정방향 순회(ws→pane→tab)를 신규 작성한다. A의 단일 IA + 검증된 status 파이프라인 그대로 재활용해 신규 결함 표면 최소. 트리 빌더가 join을 신규 작성하므로 AgentDashboardViewModel은 보존 대상이 아니라 폐기 후보다(보존되는 것은 데이터 인프라, 코드 join 로직이 아님).
선택: 옵션 C (트리 우선 + 데이터 레이어 보존)
  • master-plan-v2의 "진화" 의도와 P3.5 D3-7의 폐기/보존 매핑 둘 다와 정합한다. D3-7은 UI 컴포넌트(카드/뷰모델/정렬/점프)는 폐기, 데이터 인프라(observer/coordinator/badge/index/viewport provider)는 보존으로 이미 명확히 갈라 두었다.
  • 옵션 B의 병존은 P3.5 R8(사이드바 ↔ 트리 navigation 중복)을 더 악화시킨다. 한 화면에 그리드+트리+사이드바 3중 navigation은 사용자 혼란이 명백하다.
  • 옵션 A 대비 차이는 데이터 인프라(observer/coordinator/badge/SessionIndex)를 그대로 재활용해 검증된 status 파이프라인 위에서 트리 빌더만 신규 작성한다는 점. 신규 결함 표면을 줄인다.
  • 미출시 기능이므로 카드 그리드 폐기에 따른 실사용자 회귀는 없다(현재 구식 카드 그리드가 wired된 채 출시되지 않았다).

무효화 사유: 옵션 B는 navigation 삼중화 + 의미 충돌로 무효화(master "진화" 의도와 배치). 순수 옵션 A(데이터 로직까지 전부 폐기)는 검증된 status 파이프라인을 버려 신규 결함 위험을 키우므로 C에 흡수하여 부분 채택.

2.4 Viable Options — 스코프 축: Day 0(빌더 추출) 포함 vs 제외

lazy 미생성 탭을 우측에서 어떻게 다룰지에 따라 Day 0(buildCommand 공용 추출)의 필요 여부와 전체 일정이 갈린다. 2개 옵션을 비교한다.

옵션내용일정장점단점
D0-keep. Day 0 포함 우측 클릭 시 미생성 탭이면 TerminalCommandBuilder.build로 신규 spawn. buildCommand를 Sources/Session/으로 공용 추출. 4일 master § 3 제약 2 요구("클릭 시점 생성 + buildCommand 공용 추출 선행")를 정확히 충족. 모든 tab이 우측에서 즉시 동작. 부팅 직후 전 탭 lazy 시나리오에서도 우측 정상. 추출 리팩터링 1일 + 빌더 배치/타겟 정합 필요.
D0-drop. Day 0 제거 우측 클릭 시 미생성 탭이면 신규 spawn 대신 disabled placeholder("이 세션을 워크스페이스 탭에서 먼저 여세요") 표시. 추출 불필요. 3일 Day 0 삭제로 일정 압축. 빌더 배치 우려 자동 해소. spawn 부작용 회피. master § 3 제약 2 위반. 부팅 직후 모든 탭이 lazy이므로 우측이 대부분 placeholder → "절반만 동작하는 UI" 회귀로 보임. 세션 공유의 "어느 탭이든 우측에서 작업" 핵심 가치 훼손.
선택: 옵션 D0-keep (Day 0 포함, 4일 유지)
  • master-plan-v2 § 3은 제약 2를 "한 번도 열린 적 없는 탭은 surface가 lazy 미생성 — 클릭 시점 생성(이때는 신규 spawn)이며 PaneTerminalView.buildCommand의 공용 추출이 선행 리팩터링"이라고 명문화했다. lazy spawn과 buildCommand 추출은 master가 정한 MVP 동작이지 선택 사항이 아니다.
  • 부팅 직후에는 모든 탭의 surface가 lazy 미생성이다(P2 결정: 자동 respawn 안 함). D0-drop을 택하면 첫 진입 시 우측이 거의 항상 placeholder가 되어 핵심 가치가 깨진다.
  • 빌더 배치 타겟 정합은 Day 0를 없애는 게 아니라 빌더를 Sources/Session/(단위 테스트 타겟 공통 멤버십 + 의존 타입 동일 타겟)에 두어 해소한다. Day 0 유지의 유일한 비용이던 "추출이 테스트 불가 위치에 갇힘"은 제거된다.

무효화 사유: D0-drop은 master § 3 제약 2를 정면 위반하고 첫 진입 UX를 훼손하므로 무효화. placeholder는 fallback(Day 2 스파이크 실패 시 read-only 미러)에서만 부분 등장하며, 정상 경로의 기본 동작이 되어선 안 된다.

3. Day-by-Day Work Package

총 4일. 각 Day는 자체 종료 조건(측정 가능)을 갖는다. Day 종료마다 회귀 검증 후 commit한다(Day별 commit 관례).

빌드 시스템

이 레포는 SPM이 아니라 XcodeGen(project.yml) + xcodebuild다(Package.swift 부재). 모든 종료 조건은 다음 형식을 따른다(docs/plans/p5/p5-detailed.html:540/575 형식 차용).

  • 빌드: xcodegen generate (exit code 0) → xcodebuild build -scheme ClaudeAlarmTerminal (exit code 0)

Day 2 — 우측 호스트: AgentTerminalHostView (재부모화 스파이크 선행)

목표
선택된 tabId의 기존 surface를 acquireExisting으로 mount하는 NSViewRepresentable을 만든다. 같은 PTY/scrollback 공유 + 재부모화 규율(워크스페이스 탭 복귀 시 뷰 반환) + lazy 미생성 탭 클릭 시 신규 spawn. 재부모화는 미검증 가정이므로 스파이크로 먼저 실증한다.
재부모화는 미검증 가정 — 스파이크 게이트 필수

acquireExisting의 실제 사용처는 GhosttyViewportProvider(viewport 텍스트 read), AgentJumpAction(makeFirstResponder focus), InternalInputWiring(surfaceHandle 취득) 3곳뿐이고, 반환된 NSView를 다른 SwiftUI 컨테이너에 mount하여 재부모화하는 경로는 코드베이스에 전무하다. 워크스페이스 탭 전환은 같은 컨테이너 트리 내 .id() 재생성이지 다른 부모로의 이동이 아니다. 따라서 "본 순간 워크스페이스 탭에서 사라지고 복귀 시 돌아온다"는 동작은 가정이며 Day 2 스파이크가 실증 게이트다.

산출물
  • (스파이크 선행) 재부모화 GUI 스파이크 — 최소 prototype으로 acquireExisting NSView를 두 컨테이너 사이에서 이동시켜 scrollback/포커스 보존을 육안 실증. docs/plans/p5.5/walkthrough.md에 결과 기록(성공/실패 + 관측). 실패 시 fallback 경로로 전환.

  • release graceful 전환(엣지): 우측 mount 중인 tab의 세션이 .exited 되거나 closeTab/closeWorkspace로 surface가 release되면 우측 호스트는 EmptyState로 graceful 전환한다. AgentSplitView가 selectedTabId의 surface 존재 여부(registry.contains(id:))와 snapshot .exited를 관찰하여 호스트 대신 EmptyState를 렌더한다(Day 3 조립 시 wire, 종료 조건은 Day 3 A11).
기술 노트
  • 재부모화 핵심(가정, 스파이크로 검증): acquireExisting이 반환한 NSView가 이미 다른 곳에 mount된 경우, SwiftUI representable이 이를 자신의 컨테이너에 넣으면 AppKit removeFromSuperview + addSubview가 자동 발생할 것으로 기대한다. 스파이크가 이 동작과 scrollback 보존을 확인해야 본 설계가 성립한다(제약 1).
  • lazy spawn 경로: 한 번도 안 열린 탭은 acquireExisting이 nil → acquire(id:factory:)로 신규 생성. 이때 PTY가 새로 spawn된다(제약 2, 신규 셸이지만 같은 tabId라 이후엔 공유됨).
  • 우측 host frame이 변하면 같은 PTY로 SIGWINCH(제약 3). 워크스페이스 복귀 시 재리사이즈됨. GhosttyTerminalView.setFrameSize/layout()이 이미 처리(기존 코드 재사용).
  • surfaceHandle 접근은 GhosttyViewportProvider 패턴(ADR-E alloc/free 1:1)을 침해하지 않음 — 본 호스트는 read_text를 호출하지 않고 view를 mount만 함.
  • fallback (스파이크 실패 시): 재부모화가 scrollback 손상/포커스 유실/렌더 깨짐을 일으키면, 우측 호스트를 read-only 미러로 격하한다 — GhosttyViewportProvider.readViewportText로 선택 tab의 현재 viewport를 주기 폴링 표시하고(기존 ViewportPollingTimer 재사용), 실제 입력은 "이 세션은 워크스페이스 탭에서 조작하세요" 안내 + 워크스페이스 탭 점프 버튼으로 위임한다. 세션 공유의 "라이브 입력"은 포기하되 "라이브 관찰 + 빠른 점프"는 보존된다. fallback 채택은 Risks 첫 행에 바인딩.

enum TerminalCommandBuilder {
    /// 특정 tab 의 kind 로 실행 command 문자열을 합성한다.
    /// 형태: /usr/bin/env KEY=VAL /bin/zsh -l (POSIX env(1) 호환).
    /// 동작은 기존 PaneTerminalView.buildCommand 와 byte-identical 이어야 한다.
    /// tab 을 직접 받는 단일 시그니처. activeTab fallback 은 호출부(위임 스텁)가 담당.
    static func build(workspace: Workspace, pane: Pane, tab: Tab) -> String {
        switch tab.kind {
        case .claude:
            // R2 이유 보존: zsh -lic 'exec claude' 로 fnm/nvm PATH init.
            let claudePath = (try? resolveClaudeBinary()) ?? "claude"
            let shell = workspace.envSnapshot["SHELL"] ?? "/bin/zsh"
            return "\(shell) -lic 'exec \(shellQuote(claudePath))'"
        case .shell:
            // HISTFILE 격리 path. 실패 시 빈 문자열 fallback (UI 가 죽지 않도록).
            let dir = (try? SessionSpawnEnv.zshHistoryDir(
                workspaceId: workspace.id, paneId: pane.id)) ?? ""
            let shell = workspace.envSnapshot["SHELL"] ?? "/bin/zsh"
            return "/usr/bin/env HISTFILE=\(shellQuote(dir + "/history")) \(shell) -l"
        }
    }

    /// 공백/특수문자 path 는 단일 인용부호로 quote. 기존 동작 보존.
    static func shellQuote(_ path: String) -> String { /* 기존 로직 이전 */ }
}

// Sources/UI/AgentView/AgentTreeNode.swift (신규)
enum AgentTreeNode: Identifiable {
    case workspace(id: UUID, name: String, children: [AgentTreeNode])
    case pane(id: UUID, position: PanePosition, children: [AgentTreeNode])
    case tab(id: UUID, sessionId: UUID?, name: String, kind: PaneKind,
             snapshot: SessionStatusSnapshot?)   // status 뱃지 source

    var id: UUID { /* 각 case 의 id */ }
    var children: [AgentTreeNode]? { /* workspace/pane 만 non-nil, tab leaf 는 nil */ }
    var selectableTabId: UUID? { /* tab case 일 때만 id, 나머지 nil */ }
}

// Sources/UI/AgentView/AgentTreeBuilder.swift (신규, 순수 함수)
enum AgentTreeBuilder {
    /// workspaces × snapshots 를 3단 트리로 join. agentView kind 는 제외.
    /// 정렬: workspace 추가순 (P3.5 Q-β). 정방향 순회(ws → panes → tabs) + snapshot lookup.
    static func build(
        workspaces: [Workspace],
        snapshots: [UUID: SessionStatusSnapshot]
    ) -> [AgentTreeNode] {
        workspaces
            .filter { $0.kind == .normal }
            .map { ws in
                let paneNodes = ws.panes.map { pane in
                    let tabNodes = pane.tabs.map { tab in
                        AgentTreeNode.tab(
                            id: tab.id, sessionId: tab.sessionId,
                            name: tab.name, kind: tab.kind,
                            snapshot: tab.sessionId.flatMap { snapshots[$0] }
                        )
                    }
                    return AgentTreeNode.pane(
                        id: pane.id, position: pane.position, children: tabNodes)
                }
                return AgentTreeNode.workspace(
                    id: ws.id, name: ws.name, children: paneNodes)
            }
    }
}

// Sources/UI/AgentView/AgentTreeSelection.swift (신규)
@MainActor
final class AgentTreeSelection: ObservableObject {
    @Published var selectedTabId: UUID?
    @Published var expandedNodeIds: Set<UUID> = []

    /// 트리의 첫 sessionId 보유 tab 을 선택 (추가순 first). 전부 비면 nil.
    func selectFirstAvailable(workspaces: [Workspace]) {
        for ws in workspaces where ws.kind == .normal {
            for pane in ws.panes {
                for tab in pane.tabs where tab.sessionId != nil {
                    selectedTabId = tab.id
                    expandedNodeIds.insert(ws.id)
                    expandedNodeIds.insert(pane.id)
                    return
                }
            }
        }
        selectedTabId = nil   // EmptyState 트리거
    }

    /// 사이드바 클릭 동기화: workspace expand 만, selectedTabId 무변경 (R8 규칙).
    func syncFromSidebar(workspaceId: UUID) {
        expandedNodeIds.insert(workspaceId)
    }
}

6. ADR 운영화 — ADR-I (1 surface = 1 NSView)

ADR-I(SurfaceRegistry가 모든 surface의 owner, hibernate/pause 0줄)를 본 phase의 구체 정책으로 적용한다. 근거: docs/p2/adr-i-application.md.

ADR-I 원칙P5.5 적용 정책
1 surface = 1 NSView (tabId 1:1) agent-view 우측 호스트는 기존 surface가 있으면 신규 생성하지 않는다. acquireExisting(id: tab.id)로 기존 NSView를 그대로 mount, lazy 미생성 시에만 acquire factory로 1회 생성. 두 번째 GhosttyTerminalView 직접 생성 경로 부재는 Day 2 종료 조건 grep -cE "GhosttyTerminalView\(" Sources/UI/AgentView/AgentTerminalHostView.swift 결과 1(factory 내부 1건만)로 봉쇄.
registry가 lifetime owner 우측 호스트는 surface를 release하지 않는다. agent-view를 떠나도 surface는 워크스페이스 탭/registry에 alive. release는 기존 WorkspaceCoordinator.closeTab/closePane/closeWorkspace만 수행(무변경).
동시 미러링 불가 같은 tabId surface를 우측 호스트와 워크스페이스 탭에 동시 표시 불가. AppKit addSubview가 마지막 컨테이너로 강제 이동(제약 1). 본 정책을 AgentTerminalHostView 주석 + 수락 기준 A1로 명문화.
lazy 미생성 surface 한 번도 안 열린 탭은 acquireExisting nil → acquire factory로 1회 신규 spawn(제약 2). 이때 TerminalCommandBuilder(Day 0 추출)로 spawn command 일관성 보장.
리사이즈 공유 우측 host frame이 워크스페이스 탭과 다르면 같은 PTY로 SIGWINCH(제약 3). GhosttyTerminalView의 기존 setFrameSize/layout() 콜백 재사용. 신규 리사이즈 코드 0줄.
alloc/free 1:1 (ADR-E 인접) 우측 호스트는 ghostty_surface_read_text를 호출하지 않으므로 GhosttyViewportProvider의 alloc/free 1:1 불변식과 무관. mount만 수행.

8. Risks & Mitigations
