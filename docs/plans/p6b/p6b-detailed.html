1. 개요

Goal

P6a가 발급한 디바이스 토큰에 수명과 폐기 경로를 부여하고, loopback에 갇혀 있던 WS endpoint를 Tailscale tailnet에 노출한다. 구체적으로 (1) 30일 만료 강제 + 만료 7일 전 데스크톱 UI 경고, (2) 설정 화면 디바이스 목록에 수동 revoke 배선 + "Device lost" 즉시 무효화, (3) revoke 시 이미 살아 있는 WS 연결을 즉시 끊기, (4) tailscale status 사전 진단 + 한국어 안내 후 WS endpoint를 Tailscale IP에 바인딩, (5) P6a에서 누적 부채로 남은 "미사용 페어링 토큰"을 claim 시 승격 모델로 정리한다. 모바일 클라이언트 없이 DaemonDevCLI 확장과 단위/통합 테스트로 만료·revoked 토큰의 WS 연결·push가 모두 즉시 거부됨을 측정 가능하게 검증한다.

2. RALPLAN-DR Summary

2.1 Principles (설계 원칙)

  1. tailnet은 1차 경계, Bearer는 2차 방어 — WS endpoint를 Tailscale IP에 바인딩해 tailnet 밖에서는 포트 자체가 보이지 않게 하고(1차), 그 위에서 P6a의 Bearer 2단 게이트가 디바이스를 인증한다(2차). 두 방어는 독립이라 어느 한 층이 뚫려도 다른 층이 남는다. 0.0.0.0 노출 후 인증에만 의존하는 단층 방어는 거부한다(D-1).
  2. 폐기는 즉시·완전해야 한다 — revoke는 (a) store에서 토큰 무효화, (b) 살아 있는 WS 연결 즉시 끊기(같은 deviceId 동시 N연결 전부), (c) push 거부(seam)를 모두 포함한다. "다음 메시지에서 거부"는 누출 창을 남기므로 즉시 끊기를 채택한다(D-2). revoke 경로는 DeviceRevocationCoordinator 한 곳으로 수렴해 UI·CLI·"Device lost"가 같은 순서 보장 다단계 절차를 거친다(원자 트랜잭션은 아니나 store.revoke 선행이 사이 구간 재연결을 verifier가 거부하므로 안전 — §5.4).
  3. 토큰은 claim된 것만 수명을 갖는다 — 코드 발급만 하고 claim 안 된 토큰이 30일 누적되는 P6a 부채를 구조적으로 제거한다. 발급 시엔 pending(코드 ttl=5분 수명)으로 두고, claim 성공 시에만 30일 active로 승격한다(D-3). 미claim 토큰은 코드 만료와 함께 자연 소멸한다.
  4. Tailscale 미충족은 silent하지 않게 — Tailscale 미설치/미로그인/오프라인은 데몬을 조용히 loopback으로 떨어뜨리는 대신 진단 결과를 한국어로 명시 노출하고, 사용자가 상태를 알 수 있게 한다. 진단은 TailscaleProbing seam으로 격리해 fake로 전 분기를 테스트한다.
  5. 모바일 부재를 존중하는 미배선 정책 — 만료 7일 전 "알림"은 데스크톱 UI callout으로 실배선하고, push 알림과 fetchHint는 seam·명세만 두고 코드를 배선하지 않는다(모바일 수신부가 P10a/P10b). EnvelopeKind fetch 추가는 P7 freeze 대상으로 기록만 한다(v0.9에 손대지 않음).

2.2 Decision Drivers (Top 3)

  1. tailnet 노출의 바인딩 불확실성 — NWListener의 requiredLocalEndpoint를 Tailscale 100.x 주소로 바꿨을 때 실제로 그 인터페이스에만 바인딩되는지, IP 변동(로그인/로그아웃) 시 재바인딩 경로가 무엇인지, tailscale CLI 출력 파싱이 환경마다 안정적인지가 SDK·외부 도구 의존이라 불확실하다. Day 0 스파이크(실측 바인딩 + CLI 진단)가 D-1 확정의 1차 분기다.
  2. 폐기 실효성 0 누출 (deliberate) — revoke가 store만 바꾸고 살아 있는 연결을 끊지 못하면 "분실 디바이스"가 다음 침묵 동안 계속 통신한다. WS는 long-lived 연결이라 "다음 envelope에서 거부"의 누출 창이 무한정일 수 있다(클라이언트가 envelope을 안 보내면 영원히 생존). 즉시 끊기 메커니즘의 정확성을 1급 종료 조건으로 둔다.
  3. 기존 383 green 비파괴 + P6a seam 정합 — P6a가 revoke(id:) seam, expiresAt/revoked 스키마, verifier 거부 로직을 이미 두었다. P6b는 이들을 깨지 않고 배선만 더해야 한다. requiredLocalEndpoint 변경은 DaemonTests의 loopback 가정(A8 lsof, 라운드트립)을 건드릴 수 있으므로 바인딩 전략을 기본 loopback으로 두고 tailnet은 opt-in으로 한다.

5.1 DeviceLifecyclePolicy — pending/active 전이 + 만료 임박 (Sources/Pairing/)

// 토큰 수명 정책. P6a Device.expiresAt 스키마 위에 "발급=pending, claim=active 승격"
// 의미론을 얹는다. now 주입 clock으로 만료 경계를 결정론적으로 테스트한다(PairingSession 선례).
public struct DeviceLifecyclePolicy: Sendable {
    let activeLifetime: TimeInterval     // 기본 30일(부록 A CLAUDE_ALARM_DEVICE_TOKEN_EXPIRY_DAYS)
    let expiringWithinDays: Int       // 기본 7일 — 이 안이면 "만료 임박"

    // pending 만료: 코드 ttl(5분)을 expiresAt로 둔다. claim 안 되면 verifier가 5분 후 거부.
    // "발급=권한 부여"가 아니라 "claim=권한 부여"라는 의미론을 강제하는 핵심(D-3).
    public func pendingExpiry(codeTTL: TimeInterval, now: Date) -> Date {
        now.addingTimeInterval(codeTTL)
    }

    // claim 성공 시 active 승격: expiresAt를 now+30일로 갱신. 동일 deviceId 재upsert는
    // P6a upsert가 옛 tokenId/secret 폐기와 정합한다.
    public func activeExpiry(now: Date) -> Date {
        now.addingTimeInterval(activeLifetime)
    }

    // 만료 임박 판정: 0 < (expiresAt - now) <= 7일. 이미 만료(≤0)는 제외(거부는 verifier 몫),
    // 7일 초과도 제외. 데스크톱 UI가 이 결과로 "N일 후 만료" 경고를 그린다(push는 seam·미배선, D-4).
    public func isExpiringSoon(_ device: Device, now: Date) -> Bool {
        let remaining = device.expiresAt.timeIntervalSince(now)
        let window = TimeInterval(expiringWithinDays) * 86400
        return remaining > 0 && remaining <= window
    }

    // 남은 일수(올림). UI 뱃지 "N일 후 만료" 표기용. 음수/0은 호출 전 isExpiringSoon이 거른다.
    public func daysRemaining(_ device: Device, now: Date) -> Int {
        let remaining = device.expiresAt.timeIntervalSince(now)
        return Int((remaining / 86400).rounded(.up))
    }
}

5.2 pending 발급(UI) + 데몬 레이어 승격 (부채 해소, D-3)

부채 지점 — 현행 issueNewCode

현재 PairingModel.issueNewCode는 발급 즉시 expiry = Date() + 30일로 store.upsert한다(Sources/UI/Pairing/PairingModel.swift:48~84, 30일 부여는 line 51 deviceTokenLifetime(), upsert는 line 59). claim 안 된 코드도 30일 유효 디바이스로 store에 남아 누적된다(PR #4 Copilot 리뷰 이연 부채). 아래가 교체안이다.

승격 주체 — 데몬 레이어 전용 (UI dead path 금지)

claim 소비는 데몬측에서만 일어난다 — WSServer.handlePairingClaim → PairingSession.claim이고, --pair 경로도 PairingClaimClient가 데몬에 코드를 제출한다. UI @MainActor PairingModel은 claim을 소비하지 않으므로, 승격 hook을 PairingModel에 두면 호출 경로가 전무한 dead path가 되어 정상 claim된 디바이스가 5분 후 만료된다. 따라서 승격 책임을 데몬 레이어 전용 DevicePromotionCoordinator(actor)로 1점 수렴한다 — revoke의 DeviceRevocationCoordinator(actor)와 대칭이며, 핵심 동작도 store.promote(id:to:) ↔ store.revoke(id:)로 모두 store 레이어 원자 메서드라 대칭이 완성된다(promote read-modify-write가 store actor 메서드 안에 묶여 revoke와 직렬화 → revoked 부활 차단, §5.2 하단). UI는 기존 refreshDevices() polling으로 승격 결과를 따라온다(역방향 통지 채널 신설 금지 — P6a claim/운영 채널 분리 보존).

// 발급(UI측 PairingModel): pending 디바이스(코드 ttl=5분 수명)로 upsert. 30일 부여를 claim까지 미룬다.
public func issueNewCode() async {
    do {
        let issued = try DeviceTokenIssuer.issue()
        let now = Date()
        // 변경점: now + 30일 → now + session.ttl(5분). pending 수명.
        let pendingExpiry = lifecycle.pendingExpiry(codeTTL: session.ttl, now: now)
        let deviceId = UUID()
        let device = Device(id: deviceId, name: "페어링 대기 디바이스",
                            tokenId: issued.tokenId, expiresAt: pendingExpiry)
        try await store.upsert(device, secret: issued.secret)
        // 시그니처 보강: session.issue(payload:deviceId:) — PairingSession이 (code → deviceId)
        // 매핑을 actor 내부에 보유하게 한다. payload에는 deviceId가 없어 도출 불가하므로 함께 전달.
        let code = try await session.issue(payload: payload, deviceId: deviceId)
        // ... 기존 payload 생성 + QR/코드/카운트다운 노출 (UI는 승격을 직접 트리거하지 않음) ...
    } catch { self.errorMessage = "페어링 코드 발급에 실패했습니다." }
}
승격은 store 레이어 원자 메서드 — revoked 부활 차단

승격을 Coordinator에서 find → upsert(read-modify-write)로 하면, revoke와 쓰기-쓰기 경쟁 시 폐기된 디바이스가 30일 active로 부활한다(read 시점엔 미revoked였다가 그 사이 revoke가 끼어들고, upsert가 revoked: false로 덮어씀 — Principle 2/3 정면 위반). 근본 원인은 비대칭이다: revoke는 store 동작 메서드(actor 내부 원자)인데 promote만 Coordinator의 비원자 read-modify-write였다. 따라서 promote를 DeviceStore.promote(id:to:) store 동작 메서드로 올려 revoke와 같은 원자성 수준에 둔다 — revoked 디바이스는 no-op(승격 거부)으로 명세한다. read-check-write가 store actor 메서드 한 호출 안에서 일어나 revoke와 직렬화된다.

// DeviceStore protocol에 promote(id:to:)를 revoke(id:)와 대칭인 동작 메서드로 추가한다.
// 의미: revoked 디바이스는 no-op(승격 거부), 미revoked면 expiresAt만 갱신(secret/메타 보존).
// read-check-write가 메서드 1개 안에 묶여 revoke와 같은 원자성 수준이 된다.
public protocol DeviceStore: Sendable {
    // ... 기존 list/upsert/find(byTokenId:)/secret/revoke(id:)/remove(id:) (P6a 무변경) ...
    func promote(id: UUID, to expiresAt: Date) async throws   // 신규 — revoked면 no-op
}

// InMemoryDeviceStore(actor): 메서드 내 read-check-write가 자연 원자다.
// guard !revoked로 폐기 디바이스를 거부한 뒤 expiresAt만 갱신(secret 보존 — 재발급 아님).
public func promote(id: UUID, to expiresAt: Date) async throws {
    guard let tokenId = tokenIdByDeviceId[id],
          let device = devicesByTokenId[tokenId],
          !device.revoked else { return }   // 미존재/revoked = no-op(부활 차단)
    devicesByTokenId[tokenId] = Device(id: device.id, name: device.name, tokenId: device.tokenId,
                                       fcmToken: device.fcmToken, apnsToken: device.apnsToken,
                                       expiresAt: expiresAt, revoked: false)   // !revoked 확인 후이므로 false 안전
}

// KeychainDeviceStore: revoke(id:)와 같은 패턴 — tokenId(forDeviceId:) → find → revoked 확인 →
// SecItemUpdate를 한 메서드 안에 묶는다. revoked면 update 없이 return(no-op).
// (P6a KeychainDeviceStore.revoke가 이미 SecItem read→update를 메서드 내에 묶은 선례와 동형.)

// 승격기(데몬 레이어 전용): DeviceRevocationCoordinator와 대칭. actor로 격리해 promotedCount를
// 어서션 가능한 상태로 둔다(@Sendable onClaimed 콜백이 actor를 캡처해 await로 변경 — struct였다면
// mutating이 복사본에 갇혀 카운터를 검증할 수 없다). UI는 경유하지 않는다.
public actor DevicePromotionCoordinator {
    let store: any DeviceStore
    let lifecycle: DeviceLifecyclePolicy
    public private(set) var promotedCount = 0   // actor 격리 카운터(테스트 어서션 가능)

    // pending → active 승격: store 원자 메서드 1줄 위임. deviceId는 PairingSession이 보유한
    // (code → deviceId) 매핑이 onClaimed로 넘긴다(payload에서 도출 불가). revoked면 store가 no-op.
    public func promote(deviceId: UUID) async {
        try? await store.promote(id: deviceId, to: lifecycle.activeExpiry(now: Date()))
        promotedCount += 1   // 호출 카운트(승격 시도). 실제 갱신 여부는 store no-op 정책이 판정
    }
}

// PairingSession 보강: (code → deviceId) 매핑 보유 + claim 성공 시 onClaimed(deviceId:) 발화.
// issue 시그니처에 deviceId 추가, claim 성공 분기에서 콜백 호출. DaemonBootstrap이 onClaimed를
// DevicePromotionCoordinator.promote에 바인딩한다(데몬 레이어 내부 — UI 무경유). 콜백은 actor
// 메서드를 부르므로 await가 필요하다(onClaimed: @Sendable (UUID) async -> Void).
public func issue(payload: PairingPayload, deviceId: UUID) throws -> String {
    let code = try SixDigitCode.generate()
    active = ActiveCode(code: code, payload: payload, deviceId: deviceId,   // deviceId 보유
                        expiresAt: now().addingTimeInterval(ttl), failedAttempts: 0)
    return code
}
// claim 성공 분기(기존 1회 소비 직후): await onClaimed?(entry.deviceId) → Coordinator.promote.

5.3 WSServer revoke 즉시 끊기 (D-2 옵션 A)

핵심 메커니즘 — deviceId 역참조 + 능동 cancel

P6a는 authState[clientId] → VerifiedDevice(여기에 deviceId 포함)만 둔다. revoke 대상은 deviceId인데 connection은 clientId로 색인되므로, deviceId → clientIds 역인덱스가 없으면 "그 디바이스의 살아 있는 연결들"을 찾을 수 없다. 옵션 B(다음 envelope 거부)는 이 역참조를 피하지만 누출 창이 무한정이다(long-lived 연결이 envelope을 안 보내면 영원히 생존). 옵션 A는 승격 시점에 역인덱스([deviceId: Set<clientId>] — 같은 deviceId 동시 N연결 수용)를 채우고 revoke 시 그 집합 전체를 능동 cancel()한다 — 누출 창 0.

다중 연결 — 같은 deviceId의 동시 N연결

같은 디바이스(같은 Bearer)가 동시에 2개 이상 연결할 수 있다(예: 모바일 앱 + --lifecycle 진단). 역인덱스가 [UUID: UUID] 단일 매핑이면 두 번째 승격이 첫 clientId를 덮어써, revoke 시 첫 연결이 끊기지 않고 살아남는다(Principle 2 누출 창 0 위반). 따라서 [UUID: Set<UUID>]로 두어 한 deviceId의 모든 연결을 보유하고, disconnectDevice는 집합 전체를 순회 cancel한다.

// 신규 역인덱스: 한 deviceId가 여러 clientId를 가질 수 있으므로 Set으로 보유한다.
// 승격(게이트 ②) 시 집합에 insert, handleDisconnect에서 해당 clientId만 remove.
private var deviceToClients: [UUID: Set<UUID>] = [:]   // deviceId -> {clientId...}
private(set) var disconnectedCount = 0   // testable 카운터 — A5가 "연결이 있었으면 끊김"을 측정

// handleMessage 게이트 ② 승격부에 1줄 추가(나머지는 P6a 무변경):
//   authState[clientId] = verified
//   deviceToClients[verified.deviceId, default: []].insert(clientId)   ← 추가(덮어쓰기 아님)

// revoke 시 호출. 그 deviceId에 바인딩된 살아 있는 연결을 전부 능동 종료한다.
// 연결이 없으면(미연결 디바이스 revoke) no-op — store.revoke만으로 충분(재연결도 거부됨).
public func disconnectDevice(deviceId: UUID) async {
    guard let clientIds = deviceToClients[deviceId] else { return }   // 미연결 = no-op
    // 집합 전체 순회 cancel — 같은 deviceId의 동시 N연결을 하나도 남기지 않는다(누출 창 0).
    for clientId in clientIds {
        guard let connection = connections[clientId] else { continue }
        connection.cancel()   // long-lived 연결이 envelope을 안 보내도 즉시 끊긴다.
        // cancel은 비동기 .cancelled 콜백으로 handleDisconnect를 트리거하지만, 즉시성을 위해
        // 여기서도 상태를 정리한다(handleDisconnect는 멱등 — 이중 정리 무해).
        connections[clientId] = nil
        authState[clientId] = nil
        await registry.cleanup(clientId: clientId)
        disconnectedCount += 1   // 실제 끊은 연결 수 — A5 측정용
    }
    deviceToClients[deviceId] = nil   // 그 deviceId의 전 연결 제거 후 키 삭제
}

// handleDisconnect도 역인덱스 정리를 추가(연결이 스스로 끊길 때) — 해당 clientId만 제거.
// 집합이 비면 키를 지운다(빈 Set 잔존 방지). 같은 deviceId의 다른 연결은 보존된다.
private func handleDisconnect(_ clientId: UUID) async {
    if let verified = authState[clientId] {
        deviceToClients[verified.deviceId]?.remove(clientId)
        if deviceToClients[verified.deviceId]?.isEmpty == true {
            deviceToClients[verified.deviceId] = nil
        }
    }
    connections[clientId] = nil
    outboundSeq[clientId] = nil
    authState[clientId] = nil
    await registry.cleanup(clientId: clientId)
}

5.4 DeviceRevocationCoordinator — revoke 1점 수렴 (UI/CLI/"Device lost" 공용)

원자 아님 — 순서가 보장된 다단계

이 Coordinator는 3개 독립 await를 순차 실행하는 것이지 원자 트랜잭션이 아니다(Swift actor await 사이에 다른 작업이 끼어들 수 있다). 그럼에도 안전한 이유는 store.revoke 선행 때문이다: ① 이후 단계가 진행되는 사이 구간에 그 Bearer로 재연결이 시도돼도, P6a DeviceTokenVerifier.verify의 !revoked 거부가 게이트 ②에서 막는다. 즉 ②(기존 연결 끊기)가 완료되기 전이라도 새 연결은 절대 승격되지 않는다. ②는 "이미 떠 있던 연결"만 처리하면 되고, 그 사이의 신규 연결은 ①이 닫는다. 따라서 누출 창은 "①과 ② 사이에 떠 있던 기존 연결이 그 짧은 구간 동안 살아 있을 수 있음"으로 한정되며, ②가 즉시 뒤따르므로 사실상 0이다.

대칭 구현 — 두 Coordinator 모두 actor

DevicePromotionCoordinator와 DeviceRevocationCoordinator를 둘 다 actor로 둔다(진짜 대칭). 승격기는 @Sendable onClaimed 콜백이 promotedCount를 변경해야 해서 actor가 필수다(struct mutating은 콜백 캡처 복사본에 갇혀 어서션 불가). 폐기기는 상태가 없어도 actor로 맞춰 대칭을 유지한다 — 연결 끊김 수는 WSServer.disconnectedCount(actor 격리, §5.3)가 단일 출처라 폐기기에 중복 카운터를 두지 않는다. 두 Coordinator의 핵심 동작(store.promote/store.revoke)이 모두 store 레이어 원자 메서드라는 점에서 대칭이 완성된다.

// revoke의 단일 진입점. store 무효화 + 살아있는 연결 끊기 + push 거부 통지(seam)를
// 순서가 보장된 다단계로 조율한다(원자 아님). UI 폐기 버튼, CLI --lifecycle, "Device lost"가
// 모두 이 1점으로 수렴해 세 부수효과가 항상 같은 순서로 일어난다(부분 폐기로 인한 잔존 방지).
// actor로 둔다 — DevicePromotionCoordinator(승격기)와 대칭. 상태는 없으나 대칭을 위해 actor.
public actor DeviceRevocationCoordinator {
    let store: any DeviceStore
    let server: WSServer
    // push 거부 통지 seam — P10a까지 no-op 카운터. revoked 디바이스를 push 발신 대상에서
    // 제외하는 필터로 쓰인다(D-7로 발신 호출자 자체는 dead).
    let pushRevocation: any PushRevocationSink

    // 폐기 절차: ① store에서 revoked 표시 → ② 살아있는 WS 연결 즉시 끊기 → ③ push 발신 제외 통지.
    // 부분 실패 정책: throw 가능 지점은 ① store.revoke뿐이다. ①이 throw하면 ②③에 진입하지 않는다
    // (무효화 자체가 실패했으면 끊기·push 제외는 의미 없음 — 호출자가 재시도). ②③은 non-throwing이라
    // ①이 성공한 뒤에는 항상 끝까지 진행된다. 순서가 중요: ① 선행이 사이 구간 재연결도 거부되게 한다.
    public func revoke(deviceId: UUID) async throws {
        try await store.revoke(id: deviceId)               // ① 무효화(throw 가능 — 실패 시 ②③ 미진입). 재연결도 거부
        await server.disconnectDevice(deviceId: deviceId) // ② 살아있는 연결 전부 cancel(non-throwing, disconnectedCount 증가)
        await pushRevocation.markRevoked(deviceId: deviceId) // ③ push 발신 제외(non-throwing seam)
    }
}

5.5 WSServer BindStrategy + TailscaleDiagnostics (D-1 옵션 A)

// 바인딩 전략. 기본 loopback(383 green·A8 lsof 비파괴), tailnet은 부트스트랩 opt-in.
public enum BindStrategy: Sendable, Equatable {
    case loopback                  // 127.0.0.1 (P6a 동작 그대로)
    case tailscaleIP(String)        // 100.x.y.z — tailnet 한정 노출(1차 경계)

    var host: String {
        switch self {
        case .loopback: return "127.0.0.1"
        case .tailscaleIP(let ip): return ip
        }
    }
}

// makeListenerParameters가 host를 BindStrategy로 분기(P6a 게이트 ① 핸드셰이크 클로저는 무변경):
//   params.requiredLocalEndpoint = NWEndpoint.hostPort(host: NWEndpoint.Host(strategy.host), port: .any)

// Tailscale 진단 seam. 실 구현은 Process로 CLI 호출하되 이 protocol 뒤에 격리해
// 테스트는 fake로 4분기를 전부 검증한다(Tailscale 미설치 CI에서도 결정론적).
public protocol TailscaleProbing: Sendable {
    func probe() async -> TailscaleState
}

public enum TailscaleState: Sendable, Equatable {
    case running(ip: String)   // 100.x 획득 → tailscaleIP 바인딩
    case notInstalled         // CLI 부재 → loopback 폴백 + 한국어 안내
    case notLoggedIn          // BackendState=NeedsLogin → loopback + 안내
    case offline              // BackendState=Stopped 등 → loopback + 안내

    // 사용자 표시용 한국어 사유(Principle 4 — silent 실패 금지). secret/실IP 미포함.
    public var koreanReason: String {
        switch self {
        case .running: return "Tailscale에 연결되어 안전한 원격 접속이 가능합니다."
        case .notInstalled: return "Tailscale이 설치되어 있지 않습니다. 로컬 연결만 사용합니다."
        case .notLoggedIn: return "Tailscale 로그인이 필요합니다. 로컬 연결만 사용합니다."
        case .offline: return "Tailscale이 오프라인 상태입니다. 로컬 연결만 사용합니다."
        }
    }
}

// 진단 → 바인딩 전략. running(ip)만 tailnet 노출, 나머지는 loopback 폴백(사유 노출).
public func bindStrategy(from state: TailscaleState) -> BindStrategy {
    if case let .running(ip) = state { return .tailscaleIP(ip) }
    return .loopback
}

// 실 구현(seam 뒤 격리 — 단위 테스트 비대상, Day 0 스파이크 + Day 3 walkthrough만 친람).
// Process로 외부 CLI를 호출하고 출력을 파싱해 TailscaleState 4분기로 환원한다.
// 실행 파일 부재(launch throw)는 .notInstalled. BackendState로 로그인/오프라인을 가른다.
public struct ProcessTailscaleProbe: TailscaleProbing {
    let cliPath: String   // 부록 A CLAUDE_ALARM_TAILSCALE_CLI_PATH (경로만 — secret/IP 아님)

    public func probe() async -> TailscaleState {
        // 1) status --json으로 BackendState 판정. launch 실패 = 미설치.
        guard let statusJSON = try? run([cliPath, "status", "--json"]) else {
            return .notInstalled
        }
        let backend = Self.backendState(statusJSON)   // "Running"/"NeedsLogin"/"Stopped"...
        switch backend {
        case "NeedsLogin": return .notLoggedIn
        case "Running": break
        default: return .offline   // Stopped/NoState 등은 오프라인으로 묶는다
        }
        // 2) Running이면 ip -4로 100.x 1줄 획득. 획득 실패 시 offline로 보수적 폴백.
        guard let ipOut = try? run([cliPath, "ip", "-4"]),
              let ip = Self.firstTailscaleIP(ipOut) else {
            return .offline
        }
        return .running(ip: ip)   // 로그는 100.x.x.x 마스킹(A13). 실 IP는 메모리에만.
    }
}

5.5b WSServer makeListenerParameters — BindStrategy 분기 (P6a 게이트 ① 무변경)

// init 기본값 인자: strategy를 기본값으로 받아 기존 8개 호출부(DaemonBootstrap·DaemonDevCLI·
// 테스트 5종 등)가 strategy 인자 없이 무변경 컴파일되게 한다 — Day 2 "기존 함수 green"의 전제.
public init(registry: SessionBindRegistry, authGate: WSAuthGate,
            verifier: DeviceTokenVerifier, pairingSession: PairingSession? = nil,
            strategy: BindStrategy = .loopback,   // ← 신규 기본값 인자(기존 호출부 무변경)
            pendingWindow: TimeInterval = WSServer.defaultPendingWindow()) { /* ... self.strategy 저장 ... */ }

// 변경점은 host 한 곳뿐 — P6a 핸드셰이크 게이트 ①(setClientRequestHandler) 클로저는 무변경.
// 즉 tailnet 노출은 "어느 인터페이스에 바인딩하느냐"만 바꾸고, 인증 경로는 동일하다.
private static func makeListenerParameters(authGate: WSAuthGate,
                                          strategy: BindStrategy) -> NWParameters {
    let params = NWParameters.tcp
    let ws = NWProtocolWebSocket.Options()
    ws.autoReplyPing = true
    ws.setClientRequestHandler(authGate.queue) { _, headers in
        // ... P6a 게이트 ① 무변경: claim 핸드셰이크/Bearer 구조검증/nonce carry ...
    }
    params.defaultProtocolStack.applicationProtocols.insert(ws, at: 0)
    // 핵심 변경: 고정 "127.0.0.1" → strategy.host (loopback 또는 100.x).
    // tailscaleIP면 그 인터페이스에만 바인딩되어 tailnet 밖은 도달 불가(1차 경계).
    params.requiredLocalEndpoint = NWEndpoint.hostPort(
        host: NWEndpoint.Host(strategy.host), port: .any)
    return params
}
SDK/외부 도구 주의 — Day 0 스파이크로 실증

requiredLocalEndpoint를 100.x 주소로 바꿨을 때 실제로 그 인터페이스에만 바인딩되어 tailnet 밖(loopback 포함 다른 인터페이스)에서 도달 불가능한지, 그리고 tailscale status --json/ip -4 파싱이 환경마다 안정적인지가 외부 도구 의존이라 불확실하다. 위 코드의 D-1 옵션 A는 Day 0 스파이크가 이를 실증한 뒤에만 확정한다. 실증 실패 시 옵션 C(tailscale serve 프록시 — 데몬은 loopback 유지, 외부 Tailscale이 tailnet→loopback 포워딩)로 폴백한다. 어느 경우든 그 위의 Bearer 2단 게이트(2차 방어)는 P6a 그대로라 인증 강도는 보존된다 — D-1은 1차 경계(노출 범위)의 선택일 뿐이다.

5.6 PushSender revoked 발신 제외 (push 거부 검증 대상)

// P6a/P5 PushSender.send는 revoked 거부 로직이 없다(WS만 verifier 경유). push 발신 호출자
// 자체는 D-7로 dead(internal output tap 미지원)이나, "revoked 디바이스는 push 발신 대상에서
// 제외한다"는 필터 로직과 그 단위 검증은 본 단계에 둔다(Coordinator ③의 실효).
public protocol PushRevocationSink: Sendable {
    func markRevoked(deviceId: UUID) async
    func isRevoked(deviceId: UUID) async -> Bool
}

// 발신 필터(미래 발신 호출자가 쓸 seam): revoked 디바이스 target은 transport에 넘기지 않는다.
// 현재는 LifecycleE2ETests가 직접 호출해 "revoked는 발신 제외"를 측정한다(transport 미호출).
public func sendIfNotRevoked(_ env: PushEnvelope, target: PushTarget, deviceId: UUID) async {
    if await revocationSink.isRevoked(deviceId: deviceId) {
        excludedCount += 1          // 거부 카운터(P6b 신규 — 기존 PushSender.rejectedCount 패턴 차용)
        return                       // transport 미호출 — revoked 디바이스에 push 전송 안 함
    }
    await send(env, target: target)  // 기존 P5 send 경로(WS-attached skip + 4KB + transport)
}

5.7 fetchHint 명세 (코드 미배선 — §8 참조)

fetchHint는 push 수신 후 모바일이 본문을 보강하는 WS 채널(kind=fetch, messageId)이다. P6b는 EnvelopeKind에 fetch를 추가하지 않는다 — v0.9 스키마를 P7 freeze 전에 흔들지 않기 위함이다(Principle 5, D-5). 스키마·의미만 §8에 명세하고 P7 v1.0 freeze 대상으로 기록한다. 의사 코드는 §8을 참조한다.

6. ADR 운영화 — ADR-F: Tailscale prerequisite UX

동결 — 데스크톱 측 확정, P8 모바일 측 적용

master § 6 ADR 목록에서 ADR-F는 작성 단계가 P6b/P8이다. P6b는 데스크톱 측 정책(데몬이 Tailscale 상태를 어떻게 진단·노출·폴백하는가)을 확정하고, 모바일 측 설치/로그인 안내 화면은 P8이 본 ADR을 참조해 적용한다. 본 §6 본문이 그대로 docs/adr/F-tailscale-prerequisite-ux.md가 된다(Day 4).

Decision (결정)

  1. Tailscale은 system service 의존, tsnet 미사용. 데몬은 외부 tailscale CLI/데몬의 상태를 tailscale status --json/tailscale ip -4로 진단·소비만 하고, 임베디드 Tailscale(tsnet)은 도입하지 않는다.
  2. 데스크톱은 진단 4분기를 한국어로 명시 노출한다. running(ip)/notInstalled/notLoggedIn/offline 각각에 대해 설정 UI가 한국어 안내를 띄운다. silent 실패(조용한 loopback 폴백)는 금지한다(Principle 4).
  3. tailnet 미충족 시 loopback 폴백 + 노출. Tailscale IP를 못 얻으면 데몬은 loopback으로 폴백하되 그 사유를 진단으로 노출한다 — 데몬은 항상 기동하고(loopback 최소 보장), 원격 접속만 비활성된다.
  4. 모바일 prerequisite는 P8이 본 ADR 참조. 모바일 앱은 페어링 전 Tailscale 앱 설치 + 동일 tailnet 로그인을 요구하는 안내 화면을 P8에서 제공한다. 데스크톱이 발급하는 wsEndpoint가 100.x 주소이므로(P6b), 모바일이 같은 tailnet에 없으면 도달 불가하다 — 이 전제를 P8 UX가 안내한다.

Drivers (결정 동인)

  • master § 3 P6b "Tailscale은 system service 의존(tsnet 미사용). tailscale status 사전 진단 + 한국어 안내" — 명시적 요구.
  • tsnet 임베디드는 데몬에 Tailscale 인증·키 관리를 끌어와 보안 표면·바이너리 크기·유지보수를 키운다 — system service 위임이 경계를 단순하게 유지.
  • silent 폴백은 "왜 모바일이 연결 안 되는가"를 사용자가 진단 불가능하게 만든다 — 명시 노출이 지원 비용을 낮춘다.

Alternatives considered (검토한 대안)

대안내용기각 사유
tsnet 임베디드데몬이 Tailscale을 라이브러리로 내장해 자체 tailnet 노드가 됨데몬에 Tailscale 인증/키/업데이트 책임 유입 — 보안 표면·바이너리·유지보수 증가. system service 위임 대비 이득 없음(사용자는 이미 Tailscale 앱 사용)

Why chosen (선택 이유)

system service 의존 + 명시 진단 노출 + loopback 폴백의 조합이 (a) 데몬의 보안 경계를 단순하게 유지하고(tsnet 인증 책임 회피), (b) 사용자가 Tailscale 상태를 항상 알 수 있게 하며(silent 실패 제거), (c) Tailscale이 없어도 데몬은 loopback으로 최소 동작(점진적 저하)하는 세 목표를 동시에 만족한다. 모바일 측 안내를 P8로 미루는 것은 데스크톱이 먼저 wsEndpoint를 100.x로 발급하는 주체이므로(P6b), 그 전제를 데스크톱에서 확정하고 모바일이 소비하는 순서가 자연스럽기 때문이다.

Consequences (결과)

  • (+) 데몬이 Tailscale 인증/키를 다루지 않아 보안 표면이 작다.
  • (+) 사용자가 설정 UI에서 Tailscale 상태를 한눈에 확인 — 원격 연결 실패 진단이 쉽다.
  • (+) Tailscale 부재에도 데몬은 loopback으로 기동 — 데스크톱 단독 기능은 무영향.
  • (−) 모바일 원격 접속은 양측이 같은 tailnet에 있어야만 성립 — P8이 이 전제를 안내해야 하는 부담(MVP 수용).
  • (−) Tailscale IP 변동 시 데몬 재바인딩(재시작) 필요 — 진단 + 재시작 경로로 흡수(자동 재바인딩은 post-MVP).
  • (−) IP 변동 시 기존 발급된 wsEndpoint도 stale하다 — 페어링 payload는 발급 시점 IP를 고정 인코딩하므로(QR/6자리 코드 모두), 재바인딩 후에는 기존 디바이스가 옛 주소로 도달 불가하여 재페어링이 필요하다. 설정 UI가 현 바인딩 주소 변경을 노출해 사용자가 재페어링 시점을 판단한다(자동 endpoint 갱신은 post-MVP).

7. Pre-mortem + 확장 테스트 계획 (deliberate 특수 섹션)

7.1 Pre-mortem — 4 실패 시나리오

"P6b가 출시됐는데 보안 사고가 났다. 무엇이 잘못됐나?"를 역산한다.

시나리오 1 — tailnet 노출 후 인증 우회 경로

증상WS endpoint를 tailnet에 노출한 뒤, tailnet 내부의 다른 디바이스(또는 침해된 노드)가 Bearer 없이 또는 위조 Bearer로 데몬에 접근해 세션을 탈취하거나, 0.0.0.0 오바인딩으로 tailnet 밖에서도 접근 가능.
근본 원인(a) 바인딩이 의도와 달리 모든 인터페이스에 떠 tailnet 밖 노출(D-1 옵션 B 실수). (b) tailnet 노출 후 P6a 인증 게이트가 우회되거나, 핸드셰이크 게이트 ①이 tailnet 연결에 대해 약화됨.
완화 (계획 반영)D-1 옵션 A(Tailscale IP 직접 바인딩) — tailnet 밖에서는 포트 미노출(1차 경계). 그 위 P6a Bearer 2단 게이트 무변경(2차 방어). Day 0 스파이크가 "100.x 바인딩 서버에 loopback 클라이언트 도달 불가"를 실측(종료 조건). Day 2 종료: tailscaleIP 전략에서도 게이트 ①② 인증 경로가 loopback과 동일하게 동작(RoundTripTests 비파괴 + 무토큰 거부 테스트). 0.0.0.0 바인딩은 코드에 부재(BindStrategy는 loopback/tailscaleIP 2분기만).

시나리오 2 — revoke 후에도 살아있는 연결로 데이터 누출

증상분실된 디바이스를 revoke했는데, 이미 수립된 WS 연결이 끊기지 않아 그 디바이스가 데스크톱의 터미널 출력(output envelope)을 계속 수신. 모바일이 아무 envelope도 안 보내는 동안(수신만) revoked 연결이 무한정 생존.
근본 원인revoke가 store만 바꾸고 살아 있는 connection을 끊지 않음(D-2 옵션 B). WS long-lived 연결은 다음 inbound envelope이 없으면 재검증 트리거가 영영 안 옴.
완화 (계획 반영)D-2 옵션 A(즉시 끊기) — WSServer.disconnectDevice(deviceId:)가 deviceId 역참조([deviceId: Set<clientId>])로 그 디바이스의 모든 connection을 능동 cancel()(같은 Bearer 동시 N연결이 하나도 안 남게 — 단일 매핑 회귀 차단). DeviceRevocationCoordinator가 store.revoke 직후 순서 보장 호출(원자 아님 — store.revoke 선행이 사이 구간 재연결도 거부). Day 2 종료: "revoke 호출 시 connection 5초 내 .cancelled + boundSession == nil" + "같은 Bearer 2연결 모두 끊김(disconnectedCount == 2)" + "재연결도 UNAUTHORIZED" XCTAssertEqual. Day 4 e2e: --lifecycle의 REVOKE_DISCONNECTED 신호.

시나리오 2b — 폐기한 디바이스가 마침 claim 중이라 되살아남

증상분실 디바이스를 사용자가 revoke한 바로 그 순간, 그 디바이스(또는 코드를 탈취한 공격자)가 마침 pairing.claim을 진행 중이었다. claim 성공 통지(onClaimed)가 revoke 직후 도착해 승격이 폐기된 디바이스를 30일 active로 되살린다 — 사용자는 폐기했다고 믿지만 토큰이 살아 있다.
근본 원인승격이 Coordinator의 find → upsert read-modify-write(비원자)라, find 시점엔 미revoked였다가 그 사이 revoke가 끼어들고 upsert가 revoked: false로 덮어씀(쓰기-쓰기 경쟁). revoke는 store 원자 메서드인데 promote만 비원자인 비대칭이 원인.
완화 (계획 반영)승격을 store 레이어 원자 메서드 DeviceStore.promote(id:to:)로 올린다(M1 채택) — revoked 디바이스는 no-op(승격 거부). read-check-write(guard !revoked → expiresAt 갱신)가 store actor 메서드 1개 안에 묶여 revoke와 직렬화되므로, revoke가 먼저 커밋됐으면 뒤이은 promote는 revoked == true를 보고 즉시 return한다(부활 불가). Day 1 종료: "revoke 후 promote 호출 → revoked == true 유지 + expiresAt 미갱신" 경쟁 테스트(XCTAssertEqual). Principle 2/3 보존.

시나리오 3 — Tailscale 미설치/미로그인 환경의 silent 실패

증상사용자가 Tailscale 없이 앱을 실행. 데몬이 조용히 loopback으로 떨어지고 아무 안내가 없어, 사용자는 "왜 모바일 페어링이 안 되는가"를 진단할 수 없음. 또는 발급된 wsEndpoint가 도달 불가 주소라 모바일이 침묵 실패.
근본 원인Tailscale 미충족 분기에 사용자 노출이 없음(silent 폴백). 진단이 Process 실패를 삼켜 상태 불명.
완화 (계획 반영)Principle 4 + ADR-F — TailscaleDiagnostics가 4분기(running/notInstalled/notLoggedIn/offline)를 한국어 사유로 노출. Day 2 종료: fake 4분기가 각 한국어 사유 + 폴백 전략 XCTAssertEqual 4건. Day 3 종료: GUI walkthrough에 Tailscale 상태 카드 표시 확인. 데몬은 어느 분기든 loopback으로 기동(점진적 저하 — 데스크톱 단독 기능 무영향).

신뢰 경계 진화 — P6a → P6b → S1

P6a는 신뢰 경계를 코드베이스에 처음 도입하되 loopback에 가뒀다(단일 호스트 내부). 토큰은 발급되나 수명·폐기 경로가 미배선이었다. P6b는 그 경계를 두 축으로 확장한다: (1) 공간축 — loopback → tailnet(Tailscale IP 바인딩, 1차 경계 + Bearer 2차 방어), (2) 시간축 — 무기한 토큰 → 30일 수명 + 즉시 폐기(revoke 시 살아 있는 연결까지 끊기). 동시에 P6a 부채(미claim 토큰 누적)를 claim 승격 모델로 정리한다. S1은 여기서 검증된 디바이스 토큰·tailnet 경계 위에서 실 FCM/APNs wire를 한 번 통과시킨다 — P6b가 닫지 못한 push 발신 호출자(D-7, C4 의존)는 S1/P10a가 잇고, P6b는 "revoked는 발신 제외"라는 필터 규약만 미리 둔다.
