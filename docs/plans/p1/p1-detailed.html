1. 개요

Goal

AppKit 윈도우 1개 안에서 libghostty 기반 터미널 뷰가 단일 PTY(claude 또는 $SHELL)를 렌더링합니다. 다중 세션, workspace, tab은 본 단계 범위 밖이며 P2 이후로 미룹니다.

Side

desktop (macOS, Apple Silicon)

Scope (in) — 이번 단계에서 반드시 끝낸다

  • Swift + AppKit 프로젝트 부트스트랩 (Xcode project, Developer ID 서명 준비)
  • libghostty Swift 바인딩 통합 (단일 인스턴스)
  • PTY abstraction: Session { id, kind: claude|shell, ptyHandle, cwd, createdAt }
  • PTY spawn: claude CLI 또는 $SHELL을 cwd에서 실행
  • 키 입력, 리사이즈(SIGWINCH), 스크롤 wiring
  • 종료 처리: PTY EOF → 세션 상태 exited, claude의 경우 claudeSessionId를 메모리에 보존
  • 환경 변수 로딩: CHAT_TERMINAL_MAX_SESSIONS=1로 강제, CHAT_TERMINAL_WORKSPACE_ROOT는 cwd로만 사용
  • Daemon lifecycle 정책 골격: NSProcessInfo.processInfo.beginActivity(options:)로 백그라운드 작업 보호, Power Nap 스케줄 hook 지점 확보, lid-close 시 PTY 핸들(파일디스크립터 + child PID) 보존 정책을 코드 레벨에서 명세

Out of scope — 이번 단계에서 절대 손대지 않는다

  • workspace tab, agent-view (각각 P2, P3)
  • multi-pane split (P2)
  • WS 서버, pairing, Push (P4 ~ P6)
  • 세션 영속화 / resume (out of scope for entire MVP)

Key Deliverables

  1. libghostty 위에 동작하는 단일 터미널 뷰
  2. Session 모델 + SessionManager 골격(create, terminate, get 만 지원, max=1)
  3. 코드사이닝/노타라이즈 hook이 빌드 스크립트에 자리 잡음 (본 단계에서는 dry-run)

2. RALPLAN-DR Summary

Principles (5)

  1. Vertical slice principle. P1은 "엔진 시동" 단계이지만 그 자체로 한 가지 가치(claude/zsh를 GPU 가속 터미널에서 실행) 를 제공해야 합니다. 수평 레이어(UI만, PTY만)로 쪼개지 않습니다.
  2. Korean-safe UTF-8 from day 1. 한국어 입력/출력 검증을 P1 수락 기준에 포함하여 P3 agent-view까지 미루지 않습니다. 다바이트 경계 손상이 발견되면 단계 미통과로 간주합니다. Day 2의 wcwidth micro-check가 본 원칙의 Day 1 effectiveness 보장 메커니즘이며, Day 6의 full acceptance 4일 전에 조기 경고를 제공합니다.
  3. Deferral, not fallback. libghostty 통합이 timebox 안에 끝나지 않으면 SwiftTerm 등으로 우회하지 않고 P1을 재계획합니다. ADR-A C1 timebox 게이트가 명시적 결정 경로입니다.
  4. Timebox over indefinite spike. libghostty 통합 시도는 2주 timebox로 묶고, 그 안에 끝나지 않으면 자동 무효화 절차를 따릅니다.
  5. Sign & notarize dry-run early. Developer ID/keychain-profile 검증을 P1에서 미리 통과시켜 P6 release 단계의 위험을 분산합니다.

Decision Drivers (Top 3)

  1. libghostty 통합이 본 단계 최대 미지수. Swift 바인딩 안정성과 API 변경 위험이 가장 크며, 전체 P1 effort의 분산 대부분을 차지합니다.
  2. Single-developer 효과 예산. 5~8일 base + 컨틴전시 2일이 전체이므로, 결정 옵션은 "이 예산 안에서 가능한가"로 평가합니다.
  3. 한국어 UTF-8 wide-char는 P1에서 검증한다. P3 agent-view까지 미루면 두 단계가 동시 실패할 위험이 있으므로 P1 수락 기준에 포함합니다.

Viable Options Considered (3)

옵션설명장점단점판정
A. libghostty embed Ghostty 프로젝트의 라이브러리 형태(XCFramework 또는 static lib)를 Swift 바인딩으로 임베드. 2주 timebox 적용. GPU 가속, 정확한 ANSI/wide-char 처리, Mitchell Hashimoto 유지보수, 한국어 IME 검증된 코드 경로. Swift 바인딩 공식 문서 부족, API 변경 위험, XCFramework 빌드 시간. 선택 (timebox 적용)
B. SwiftTerm SwiftTerm 라이브러리 사용 (NSView 기반). Swift 네이티브 패키지, 통합 용이. GPU 가속 없음, 폰트/렌더링 품질이 libghostty 대비 낮음, 한국어 wide-char 검증 부담을 우리가 짐. 거부 — 렌더링 품질이 spec 요구를 만족하지 못함
C. NSTextView + 자체 ANSI 파서 NSTextView 위에 SGR 파서 직접 작성. 의존성 0, 완전한 통제. wide-character/한국어 처리 fragile, 터미널 충실도 낮음, 자체 파서 유지보수 비용이 P1 예산을 압도. 거부 — 단일 개발자 예산을 초과함
선택 근거
Option A는 spec이 요구하는 렌더링 품질과 한국어 정확도를 동시에 만족하는 유일한 경로입니다. API 불안정성은 commit pin과 2주 timebox(ADR-A C1)로 흡수합니다. Option B/C는 P3 agent-view, P4 ring buffer 등 후속 단계의 비용까지 증가시키므로 거부합니다.

3. Day-by-Day Work Package

  • docs/build-setup.md — Developer ID 인증서 keychain 등록 절차

예상 소요

기술 노트

Day 4 — PTY spawn 기본기 (forkpty/posix_spawn + setsid + TIOCSCTTY + EAGAIN read loop)

목표
libghostty와 독립적으로 PTY 마스터/슬레이브 쌍을 만들고, child process로 zsh를 띄워 stdout이 마스터 fd로 흘러나오는지 확인합니다.
산출물
  • Sources/PTY/PTYHandle.swift — masterFD, childPID, openpty 호출 래퍼
  • Sources/PTY/PTYSpawner.swift — posix_spawn + posix_spawn_file_actions + setsid + TIOCSCTTY
  • Sources/PTY/PTYReader.swift — dispatch_io 또는 GCD 기반 EAGAIN-aware read loop
  • 단위 테스트: zsh -c 'echo hello' spawn 후 마스터 fd에서 "hello\r\n"이 읽힌다
OMC 위임
executor (opus) — POSIX 저수준 API, signal, fd 수명 관리
예상 소요
1.0~1.5일
기술 노트
  • posix_spawn + posix_spawn_file_actions_addinherit_np로 PTY slave fd를 child의 stdin/stdout/stderr에 매핑
  • child 측에서 setsid() + ioctl(TIOCSCTTY) 호출 — POSIX_SPAWN_SETSID flag로 setsid는 처리 가능, TIOCSCTTY는 (P1 결정 항목)
  • 마스터 fd는 fcntl(F_SETFL, O_NONBLOCK)로 non-blocking 모드 설정 → EAGAIN/EWOULDBLOCK 발생 시 GCD source가 자동으로 재시도
  • EOF 감지: read return 0 → child 종료 처리 (Day 5의 SessionManager가 사용)
  • 경고: fork()는 Hardened Runtime + ARC와 충돌 위험이 있으므로 posix_spawn을 1차 선택. forkpty(3)는 fork 호출이 내포되어 있으므로 우리 쪽에서 openpty(3) + posix_spawn 조합으로 직접 구현 — (P1 결정 항목)

종료 조건

  • PTYSpawner.spawn(command: "/bin/zsh", cwd: ...)가 (masterFD, childPID)를 반환한다
  • 마스터 fd로 write한 "echo hi\n" 입력이 마스터 fd로 "hi\r\n"으로 돌아온다
  • child가 종료되면 read가 0 (EOF)을 반환하고 PTYReader가 종료 콜백을 호출한다
  • 마스터 fd가 명시적 close 호출 후 fd leak 없이 정리된다 (lsof -p 검증)
  • tcgetpgrp(masterFD)가 child PID를 반환한다 (Darwin controlling tty 획득 확인). 미일치 또는 -1 반환 시 ioctl(slaveFD, TIOCSCTTY, 0) 명시 호출 경로를 활성화하고 재검증.
  • ADR-A C1 sub-timebox Day 3/5가 종료됨이 일지에 기록되며, Day 4 surface-display gate (§6.2 신규 조건 1) 가 검증된다: 본일 종료 시 libghostty surface에 최소 1개 글자라도 그려진 상태이다.

5.1 PTY spawn (setsid + TIOCSCTTY + TIOCSWINSZ + EAGAIN read loop)

import Darwin
import Foundation

struct PTYHandle {
    let masterFD: Int32
    let childPID: pid_t
    let slavePath: String
}

enum PTYError: Error {
    case openptyFailed(errno: Int32)
    case spawnFailed(errno: Int32)
    case ttySetCtrlFailed(errno: Int32)
}

func spawnPTY(command: String, args: [String], cwd: String,
              env: [String: String], rows: UInt16, cols: UInt16) throws -> PTYHandle {
    // 1. openpty(3): allocate master/slave pair
    var master: Int32 = -1
    var slave:  Int32 = -1
    var name = [CChar](repeating: 0, count: 128)
    var ws = winsize(ws_row: rows, ws_col: cols, ws_xpixel: 0, ws_ypixel: 0)

    guard openpty(&master, &slave, &name, nil, &ws) == 0 else {
        throw PTYError.openptyFailed(errno: errno)
    }

    // 2. master fd: non-blocking + close-on-exec
    _ = fcntl(master, F_SETFL, fcntl(master, F_GETFL) | O_NONBLOCK)
    _ = fcntl(master, F_SETFD, FD_CLOEXEC)

    // 3. posix_spawn with file actions and setsid flag
    var fileActions: posix_spawn_file_actions_t? = nil
    posix_spawn_file_actions_init(&fileActions)
    defer { posix_spawn_file_actions_destroy(&fileActions) }

    posix_spawn_file_actions_adddup2(&fileActions, slave, 0)  // stdin
    posix_spawn_file_actions_adddup2(&fileActions, slave, 1)  // stdout
    posix_spawn_file_actions_adddup2(&fileActions, slave, 2)  // stderr
    posix_spawn_file_actions_addclose(&fileActions, slave)
    posix_spawn_file_actions_addclose(&fileActions, master)
    // Darwin: POSIX_SPAWN_SETSID 만으로는 controlling tty 자동 획득이 보장되지 않음.
    // 아래의 Darwin verification step (tcgetpgrp) 를 spawn 직후에 수행하여,
    // 미일치 시 child 측에서 ioctl(slaveFD, TIOCSCTTY, 0) 명시 호출 경로를 활성화한다.

    var attr: posix_spawnattr_t? = nil
    posix_spawnattr_init(&attr)
    defer { posix_spawnattr_destroy(&attr) }

    var flags: Int16 = Int16(POSIX_SPAWN_SETSID)
                       | Int16(POSIX_SPAWN_CLOEXEC_DEFAULT)
    posix_spawnattr_setflags(&attr, flags)

    var argv: [UnsafeMutablePointer<CChar>?] = ([command] + args).map { strdup($0) }
    argv.append(nil)
    defer { argv.forEach { free($0) } }

    var envp: [UnsafeMutablePointer<CChar>?] = env.map { strdup("\($0.key)=\($0.value)") }
    envp.append(nil)
    defer { envp.forEach { free($0) } }

    var pid: pid_t = 0
    let rc = posix_spawn(&pid, command, &fileActions, &attr, argv, envp)
    close(slave)  // parent no longer needs slave
    guard rc == 0 else {
        close(master)
        throw PTYError.spawnFailed(errno: rc)
    }

    return PTYHandle(masterFD: master, childPID: pid,
                     slavePath: String(cString: name))
}

// EAGAIN-aware read loop using DispatchIO
func startReader(fd: Int32, onData: @escaping (Data) -> Void,
                 onEOF: @escaping () -> Void) -> DispatchIO {
    let queue = DispatchQueue(label: "pty.reader", qos: .userInitiated)
    let io = DispatchIO(type: .stream, fileDescriptor: fd, queue: queue) { errCode in
        if errCode != 0 { NSLog("PTY DispatchIO cleanup: errno=\(errCode)") }
    }
    io.setLimit(lowWater: 1)
    io.read(offset: 0, length: Int.max, queue: queue) { done, data, errCode in
        if let data = data, !data.isEmpty {
            onData(Data(data))
        }
        if done {
            if errCode == 0 || errCode == EAGAIN {
                return  // keep reading on EAGAIN
            }
            onEOF()
        }
    }
    return io
}

// Resize: invoke TIOCSWINSZ on the master fd; the kernel delivers SIGWINCH
func resizePTY(masterFD: Int32, rows: UInt16, cols: UInt16) {
    var ws = winsize(ws_row: rows, ws_col: cols, ws_xpixel: 0, ws_ypixel: 0)
    _ = ioctl(masterFD, UInt(TIOCSWINSZ), &ws)
}

5.3 SessionManager actor skeleton

enum BinaryResolveError: Error {
    case claudeNotFound(searched: [String])
}

// PATH 검색 → /opt/homebrew/bin/claude (Apple Silicon) → /usr/local/bin/claude (Intel Mac)
func resolveClaudeBinary() throws -> String {
    let candidates: [String] = {
        var out: [String] = []
        if let path = ProcessInfo.processInfo.environment["PATH"] {
            for dir in path.split(separator: ":") {
                out.append("\(dir)/claude")
            }
        }
        out.append("/opt/homebrew/bin/claude")
        out.append("/usr/local/bin/claude")
        return out
    }()
    for p in candidates where FileManager.default.isExecutableFile(atPath: p) {
        return p
    }
    throw BinaryResolveError.claudeNotFound(searched: candidates)
}

actor SessionManager {
    enum ManagerError: Error {
        case maxSessionsReached(currentMax: Int)
        case notFound(id: UUID)
    }

    private(set) var sessions: [UUID: Session] = [:]
    private(set) var lastClaudeSessionId: String? = nil
    private var activityScope: ActivityScope? = nil
    private let maxSessions: Int

    init(maxSessions: Int) { self.maxSessions = maxSessions }

    func create(kind: SessionKind, cwd: String, rows: UInt16, cols: UInt16) async throws -> Session {
        guard sessions.count < maxSessions else {
            throw ManagerError.maxSessionsReached(currentMax: maxSessions)
        }
        let command: String = kind == .claude
            ? try resolveClaudeBinary()
            : (ProcessInfo.processInfo.environment["SHELL"] ?? "/bin/zsh")
        let handle = try spawnPTY(
            command: command,
            args: kind == .shell ? ["-l"] : [],
            cwd: cwd,
            env: ProcessInfo.processInfo.environment,
            rows: rows, cols: cols
        )
        let session = Session(
            id: UUID(),
            kind: kind,
            ptyHandle: handle,
            cwd: cwd,
            createdAt: Date(),
            status: .running,
            claudeSessionId: nil
        )
        sessions[session.id] = session

        if activityScope == nil {
            activityScope = ActivityScope(reason: "PTY session active")
        }
        return session
    }

    // claudeSessionId 추출 경로:
    // Day 5에서 PTY stdout 스트림에 다음 패턴이 등장하면 in-memory 저장.
    // 정규식: `Session ID: ([0-9a-f-]{36})` — claude CLI 첫 출력 또는 종료 직전 출력.
    // 매칭 시 updateClaudeSessionId 호출. 미매칭 채 PTY EOF 도달 시 nil 유지 (정상).
    func updateClaudeSessionId(id: UUID, claudeId: String) {
        guard var session = sessions[id] else { return }
        session = Session.with(session, claudeSessionId: claudeId)
        sessions[id] = session
    }

    func terminate(id: UUID) async throws {
        guard var session = sessions[id] else { throw ManagerError.notFound(id: id) }
        kill(session.ptyHandle.childPID, SIGTERM)
        try await Task.sleep(nanoseconds: 1_000_000_000)
        if kill(session.ptyHandle.childPID, 0) == 0 {
            kill(session.ptyHandle.childPID, SIGKILL)
        }
        close(session.ptyHandle.masterFD)
        if let claudeId = session.claudeSessionId {
            lastClaudeSessionId = claudeId
        }
        session = Session.with(session, status: .exited)  // immutable copy
        sessions[id] = session

        if sessions.allSatisfy({ $0.value.status == .exited }) {
            activityScope = nil  // RAII triggers endActivity
        }
    }

    func get(id: UUID) async -> Session? { sessions[id] }
}

6.2 자동 무효화 조건 (auto-invalidation)

다음 중 어느 하나라도 발생하면 Option A (libghostty embed)는 P1 범위 안에서 자동 무효화됩니다. 무효화는 "다른 라이브러리로 즉시 갈아탐"이 아니라 "P1 범위/접근을 ralplan 재합의로 재계획" 을 의미합니다.

  1. Day 4 종료 시점 (surface-display gate): libghostty surface가 AppKit window에 최소 1회 표시되지 않음 (어떠한 글자라도 그려지지 않음). 본 gate 미달 시 sub-timebox 의 1차 경고.
  2. Day 6 종료 시점 (end-to-end gate): libghostty surface에 PTY 출력이 end-to-end로 한 줄 이상 흐르지 않음 (ASCII 평문 한정). 본 gate 미달 시 sub-timebox 의 hard kill-switch 발동.
  3. 한국어 wide-character 렌더링이 라이브러리 레벨 결함으로 차단되어 컨틴전시 1일로 회복 불가 (Day 6 검증)
  4. libghostty 라이선스/배포 정책이 Developer ID 노타라이즈와 호환되지 않음 (라이센스 조사는 Day 1)

6.3 무효화 시 절차 (NOT a fallback)

중요
무효화가 트리거되면 다른 터미널 라이브러리로 즉시 갈아타지 않습니다. 다음 절차를 따릅니다.
  1. Day 6 종료 시 (또는 Day 4 surface-display gate 미달 시 즉시) ralplan 합의를 재소집 (Architect + Critic). 4 working days의 phase budget이 잔여한 상태에서 ralplan 재소집이 이루어지므로 새로운 P1 작업 + 재합의 시간을 충분히 확보.
  2. 무효화 사유, 시도한 경로, 발견된 제약을 정리한 무효화 리포트 작성 (Planner)
  3. 새로운 viable options를 다시 조사 — SwiftTerm / NSTextView / 다른 GPU 터미널 라이브러리 / Web-based 등 (이 단계에서 의사결정 재시작)
  4. P1 재계획 또는 마스터 계획 단계 재분해. 잔여 4d 안에 새 옵션으로 surface-display + end-to-end 까지 도달 가능한지 평가하여 P1 종료 일정 재산정.
  5. 총 지연: ralplan 사이클(1~2일) + 새로운 P1 effort. phase budget 4d 잔여 + 신규 effort 합산.

6.4 무효화하지 않는 경계 사례

  • "libghostty가 동작은 하나 폰트 1개가 깨진다" → 컨틴전시 0.5일 내 회복 가능하므로 무효화 아님
  • "libghostty가 main HEAD에서 빌드 실패" → 1~2 commit 후퇴로 회복 가능하면 무효화 아님 (Day 2 컨틴전시 내 처리)
  • "한국어 일부 폰트에서 자모 분리" → libghostty 자체 문제가 아니라 Core Text의 폰트 substitution 체인(Apple SD Gothic Neo / Noto Sans KR 우선순위) 설정 문제일 가능성이 높으므로 단독으로는 무효화 아님

6.5 Trigger 감지 및 ownership

  • 감지 주체: executor가 매일 종료 시점에 §6.2의 조건 체크리스트를 실행합니다. Day 4와 Day 6은 명시적 gate 일자이며 종료 조건 체크리스트에 surface-display / end-to-end 항목이 포함되어 있습니다.

7. Build & Sign Dry-run

Day 8 종료 시 실행되는 정확한 명령어 시퀀스입니다. 실제 Apple 노타라이즈 제출(real submission)은 P6에서 수행하며, 본 단계는 credential과 빌드 파이프라인이 사용 가능함을 입증하는 것이 목표입니다.

3-step dry-run 정의
본 단계의 dry-run은 다음 3 단계로 정의됩니다. 각 단계는 실제 명령을 실행하여 검증 가능한 부수효과를 만들어내며, trivially 성공하는 noop 명령은 포함하지 않습니다.
  1. Step 1 — Credential validation: xcrun notarytool history --keychain-profile "${NOTARY_PROFILE}" 가 0 exit code로 종료하고 401/403 응답이 없음.
  2. Step 2 — Signed archive 생성: xcodebuild archive + xcodebuild -exportArchive 가 0 exit code로 종료하고 build/export/${APP_NAME}.app 번들이 생성됨.
  3. Step 3 — Signature 정합성 검증: codesign --verify --deep --strict 통과 + spctl --assess --type execute --verbose 가 "accepted" 또는 "rejected source=Unnotarized Developer ID" 응답 (Notarize는 P6이므로 후자도 OK).

10. Exit Gate to P2

다음 모든 항목이 충족될 때 P1을 종료하고 P2 (다중 세션 + 워크스페이스 탭 + 가로 분할) 로 진입합니다.

  • §8 수락 기준 A1~A4 모두 통과
  • Day 1~8의 모든 일자별 종료 조건 통과
  • verifier 패스에서 critical / high 이슈 0건
  • SessionManager의 max=1 enforcement가 단위 테스트로 검증되어 있어 P2에서 max=20으로 확장 시 단일 상수 변경 + 다중 세션 동시성 테스트만 추가하면 되는 상태
  • SessionManager 동시성 측정 검증: (a) grep -E 'func (create\|terminate\|get\|updateClaudeSessionId)' Sources/Session/SessionManager.swift 결과 모든 public 메서드가 async 키워드 포함. (b) grep nonisolated Sources/Session/SessionManager.swift 결과 0개. (c) Day 5 단위 테스트: 2개 동시 create() 호출 시 1개 성공 + 1개 maxSessionsReached. (d) Thread Sanitizer 활성화 상태로 Day 5 통합 테스트 통과 (data race 0건).
  • claudeSessionId in-memory 보존 정책이 코드에 존재하며 P4 reconnect에서 사용 가능 (인터페이스만 노출, 동작은 P4에서 검증)
  • Daemon lifecycle hook 지점(ActivityScope, PowerEventObserver) 이 존재하며 P4 이후 실제 동작 검증을 위한 토대가 됨
  • docs/build-setup.md, docs/lifecycle-policy.md, docs/p1-acceptance-log.md 세 문서가 작성되어 있음
  • libghostty PINNED_COMMIT이 기록되어 있고 P2에서도 동일 commit으로 빌드 가능
  • ADR-A C1 timebox가 발동하지 않고 종료됨 (= Option A가 P1 범위 안에서 성공)
P2로의 인계 사항
P2에서는 SessionManager의 max를 CHAT_TERMINAL_MAX_SESSIONS (기본 20) 으로 풀고, Workspace/Pane 레이어를 SessionManager 위에 얹습니다. P1에서 actor isolation을 견고하게 설계했다면 P2 동시성 확장이 부드럽게 진행됩니다. P1의 단일 세션 코드 경로는 P2에서도 그대로 사용됩니다 (vertical slice principle 유지).
