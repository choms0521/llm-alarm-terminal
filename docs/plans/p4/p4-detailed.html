P4 아키텍처 전체가 아래 사실 위에서 갈라진다. 가정 없이 헤더/소스를 직접 읽어 확정했다.

#확증 질문결과근거 (file:line)
C1 libghostty가 surface PTY에 raw byte를 쓰는 공개 API가 있는가? printable만 가능. ghostty_surface_text(surface, const char*, uintptr_t)로 printable UTF-8 byte 주입 가능. 단 control byte(< 0x20)는 libghostty가 keycode+mods로 자체 인코딩하므로 ghostty_surface_text로 보내면 double-encode되고, ghostty_surface_key로 보내려면 byte→keycode 역매핑이 필요한데 ESC 시퀀스(다중 byte)는 단일 keycode로 환원 불가. ghostty.h:1127(text), ghostty.h:1123(key), GhosttyTerminalView.swift:412~414(control double-encode 주석), GhosttyTerminalView.swift:785(text 실사용)
C2 비표시(inactive tab) surface에도 입력 주입이 가능한가? 가능하다 (구조적으로). ADR-I(cmux 패턴)상 비가시 surface도 SurfaceRegistry owner가 보유하여 alive. ghostty_surface_text/_key는 surface 핸들만 있으면 displayLayer/focus 없이 C ABI로 호출 가능. focus는 set_focus(ghostty.h:1113)로 분리되어 입력 주입의 전제 조건이 아니다. (런타임 검증은 Day 0 spike + Day 5 종료 조건에서 측정.) SurfaceLifecycle.swift:18~46, ghostty.h:1113
C3 어떤 origin이 masterFD를 노출하는가? .external만 ptyHandle 비-nil → masterFD 노출. .internal은 ptyHandle=nil(libghostty 소유). masterFD는 spawn 시 이미 O_NONBLOCK 설정됨. Session.swift:39, Session.swift:20~27, PTYSpawner.swift:42(O_NONBLOCK), SessionManager.swift:301(external만 close)
C4 .internal 세션에 byte-faithful 출력 tap이 있는가? 없다. libghostty 런타임 콜백(ghostty_runtime_config_s)은 wakeup_cb/action_cb/clipboard/close_surface_cb만 노출 — raw output byte 콜백 부재. ghostty_surface_read_text는 렌더된 viewport grid 텍스트 스냅샷(ANSI/제어바이트 소실, polling). byte 스트림 출력은 .external PTYReader(masterFD)에서만 가능. 해결 옵션 조사는 Day 0 feasibility spike로 전진(아래 §3 Day 0). ghostty.h:1018~1027(콜백 set — output 없음), ghostty.h:1159(read_text grid 스냅샷), PTYReader.swift:41, GhosttyViewportProvider.swift:44
C5 마스터 계획의 origin enum 명세와 실제 코드가 일치하는가? 불일치. work-plan은 spawned/internal로 표기하나 실제 코드는 external/internal. 본 P4 계획은 .external을 정본으로 사용하며 enum 리네임은 P4 범위 밖. Session.swift:20~27
C6 빌드 시스템은 무엇인가? XcodeGen(project.yml) + xcodebuild. SwiftPM 아님(Package.swift 부재). 테스트는 bundle.unit-test 타깃 + scheme을 xcodebuild test로 실행. tool은 type: tool 타깃. 모든 검증 명령은 SwiftPM swift CLI가 아니라 xcodebuild/xcodegen이어야 한다. project.yml:29~290, scripts/build.sh:34~47

C4 + 실사용 origin의 설계 함의 (load-bearing)

실앱에서 WorkspaceCoordinator는 항상 createInternal만 호출한다(WorkspaceCoordinator.swift:39, 94, 177). .external은 headless verifier/test 전용(SessionManager.swift:128,183, SessionVerifier/main.swift:84,94,125). 즉 실사용 세션은 100% .internal이며 C1(control 입력 제약) + C4(출력 tap 부재)가 직접 적용된다. 이 사실이 아래 결정을 강제한다.

  1. P4 데모/acceptance의 byte-faithful 입출력 라운드트립은 .external 세션 기준이다. dev CLI 라운드트립은 SessionManager.create(...)(external, masterFD)로 spawn한 세션에 attach한다. 입력은 PTYWriter.write(masterFD, ...), 출력은 PTYReader.onData(Data) → Utf8StreamAccumulator → WS kind=output.
  2. 실사용(.internal) 경로는 P4에서 printable 입력 주입만 지원한다. 출력 스트리밍(C4)과 control byte 입력(C1)은 P4 범위에서 명시적 미지원이며, 각각 INTERNAL_OUTPUT_UNSUPPORTED / INTERNAL_CONTROL_INPUT_UNSUPPORTED 신호를 발행한다. 이 한계를 ADR로 명문화한다(§6 ADR-P4-3, ADR-P4-4).
  3. C4 해결 경로는 Day 0 feasibility spike에서 옵션을 열거하고 P7 envelope freeze 전 결정 게이트로 넘긴다. P4 골격은 external-first이되, internal 실사용 출력은 명확히 인지된 부채로 Exit Gate에 기록한다.
정직성 메모
본 계획의 R8 single-writer 큐의 본래 시나리오("데스크톱 keystroke + 모바일 WS 입력이 동시 도착")는 external에 GUI keystroke 직접 경로가 없어 P4에서 실측되지 않는다(§8 A7 caveat). 큐는 intra-queue strict FIFO만 P4에서 검증되고, cross-origin 비추월은 internal 실사용 wiring phase에서 검증된다.

2. RALPLAN-DR Summary

Principles (설계 원칙)

  1. 순서 정합성 우선. 동일 sessionId에 대한 PTY write는 어떤 origin에서 와도 단조 FIFO여야 한다. 추월은 금지(R8).
  2. 메시지 경계 무결성. ring buffer drop, codec, 스트림 분할은 항상 UTF-8 메시지 경계에서 일어난다. byte 경계 절단은 어떤 경로에서도 발생시키지 않는다. 출력 chunk 경계 재조립은 Utf8StreamAccumulator가 담당한다.
  3. origin 비대칭의 명시적 수용. .external(masterFD/actor)과 .internal(libghostty/MainActor)은 입출력 능력이 비대칭이다. 단일 추상으로 은폐하지 않고 2-path로 드러내며, 미지원 능력은 명시 신호로 통지한다.
  4. wire 안전 직렬화. seq/ackSeq는 u64지만 항상 string으로 직렬화한다. codec은 큰 값 라운드트립을 단위 테스트로 강제한다.
  5. 골격 우선, 보안 후속. P4는 loopback foreground 골격이다. 인증/암호화/노출은 P6+로 미루되, attach 정리/리소스 누수 방지 같은 lifecycle 정합성은 지금 확보한다.

Decision Drivers (Top 3)

#Driver설명
DR1로컬 keystroke 지연 vs R8 엄격 순서 보장R8은 데스크톱 keystroke도 동일 큐에 enqueue(추월 금지)를 요구한다. 그러나 .internal 로컬 keystroke는 현재 동기 MainActor 호출(actor hop 0, GhosttyTerminalView.swift:388~423)이다. 공유 큐로 funnel하면 keystroke마다 MainActor 왕복이 강제되어 지연이 유발된다. 이 긴장이 큐 설계의 심장이다.
DR2origin별 입출력 능력 비대칭(C1, C4).external만 byte-faithful 출력(PTYReader/masterFD) + raw control 입력을 가지고, .internal은 printable 입력만 + grid 폴링 출력뿐이다. P4 acceptance(byte 단위 UTF-8 경계 테스트)는 external에서만 통과 가능하므로 데모/검증 기준 세션 종류 선택이 전체 wiring을 가른다.
DR3seq 직렬화 정밀도(53-bit 손실)seq를 number로 보내면 JS 측 2^53 초과에서 손실된다. string 직렬화가 unconditional 결정이며 codec 라운드트립 테스트로 강제(2^53+1).

Viable Options — single-writer 큐 설계 (≥2 비교)

R8("동일 sessionId 2개 이상 동시 write 금지, FIFO + per-pane lock, keystroke도 추월 없음")를 만족하는 큐 후보 비교.

측면Option A — 전체 입력 funnelOption B — 로컬 fast-path + WS만 큐 경유 (generation guard)
메커니즘모든 입력을 세션별 SerialInputQueue(per-session 단일 consumer drain)로 보낸다.로컬 keystroke는 기존 동기 MainActor fast-path 유지. WS input만 큐 경유. 세션별 monotonic generation + per-pane lock으로 상대 순서 보존.

선택: Option A (전체 입력 funnel) — .external 기준 데모로 DR1 비용을 P4에서 무력화.

선택 근거
P4 acceptance/데모는 C4에 따라 .external 세션 기준이다. external은 GUI 직접 keystroke 경로가 없으므로 Option A의 keystroke-지연 비용(DR1)이 P4 범위 내에서는 발현하지 않는다. Option B의 generation/lock 정합 복잡도를 P4 골격 단계에 도입하는 것은 ROI가 낮고 race 리스크가 크다. 따라서 P4는 Option A를 단일 SerialInputQueue로 구현하되, sink 추상 경계(InputSink)를 깨끗이 유지하여 후속 phase에서 큐 재설계 없이 internal 로컬 keystroke fast-path(Option B 부분 전환)를 sink 교체만으로 도입 가능하게 한다(§6 ADR-P4-1).
Option B 무효화 근거 (P4 한정)
Option B의 유일한 이득(internal 로컬 keystroke 지연 0)은 P4 external 데모에서 측정되지 않는 반면, generation/lock 정합 복잡도는 P4에서 즉시 비용/리스크가 된다. 따라서 P4에서는 채택하지 않고 후속 phase 결정으로 이월한다.

기술 노트

측정 가능한 종료 조건

  • xcodebuild test -scheme DaemonTests -destination 'platform=macOS' -only-testing:DaemonTests/EnvelopeCodecTests exit 0, 내부에 XCTAssertEqual(decode(encode(env)).seq, 9007199254740993) (2^53+1 라운드트립).

5.1 Envelope v0.9 + codec (seq string) + 모노토닉 검증 호출지점

public enum EnvelopeKind: String, Codable, Sendable {
    case input, output
    case sessionStart = "session.start"
    case sessionExit = "session.exit"
    case sessionTerminated = "session.terminated"
    case ack, error, pause, resume
}

public struct EnvelopeActor: Codable, Sendable, Equatable {
    public let deviceId: String
    public let userId: String?
}

public struct WSEnvelope: Sendable, Equatable {
    public let seq: UInt64          // wire에서는 string
    public let ackSeq: UInt64?      // wire에서는 string?
    public let actor: EnvelopeActor
    public let kind: EnvelopeKind
    public let code: String?
    public let payload: Data        // 항상 완성 UTF-8(§7).
}

extension WSEnvelope: Codable {
    enum K: String, CodingKey { case seq, ackSeq, actor, kind, code, payload }

    public init(from d: Decoder) throws {
        let c = try d.container(keyedBy: K.self)
        let seqStr = try c.decode(String.self, forKey: .seq)
        guard let s = UInt64(seqStr) else { throw EnvelopeCodecError.malformedSeq(seqStr) }
        self.seq = s
        if let a = try c.decodeIfPresent(String.self, forKey: .ackSeq) {
            guard let av = UInt64(a) else { throw EnvelopeCodecError.malformedSeq(a) }
            self.ackSeq = av
        } else { self.ackSeq = nil }
        self.actor = try c.decode(EnvelopeActor.self, forKey: .actor)
        self.kind = try c.decode(EnvelopeKind.self, forKey: .kind)
        self.code = try c.decodeIfPresent(String.self, forKey: .code)
        let payloadStr = try c.decode(String.self, forKey: .payload)
        self.payload = Data(payloadStr.utf8)
    }

    public func encode(to e: Encoder) throws {
        var c = e.container(keyedBy: K.self)
        try c.encode(String(seq), forKey: .seq)            // UInt64 → string
        try c.encodeIfPresent(ackSeq.map(String.init), forKey: .ackSeq)
        try c.encode(actor, forKey: .actor)
        try c.encode(kind, forKey: .kind)
        try c.encodeIfPresent(code, forKey: .code)
        guard let s = String(data: payload, encoding: .utf8) else { throw EnvelopeCodecError.nonUtf8Payload }
        try c.encode(s, forKey: .payload)
    }
}

public enum EnvelopeCodecError: Error, Equatable {
    case malformedSeq(String)
    case nonMonotonicSeq(prev: UInt64, got: UInt64)
    case nonUtf8Payload
}

// 모노토닉 검증은 wire 레벨에서 per-client lastSeq를 보유한 SessionBindRegistry 가 호출(§5.4).
// 호출지점: WSServer inbound 처리 → SessionBindRegistry.ingestInbound(clientId:env:).
func validateMonotonic(_ env: WSEnvelope, lastSeq: inout UInt64) throws {
    guard env.seq > lastSeq else { throw EnvelopeCodecError.nonMonotonicSeq(prev: lastSeq, got: env.seq) }
    lastSeq = env.seq
}

5.2 Ring buffer — message-boundary drop-and-mark

public final class SessionRingBuffer {
    private var items: [WSEnvelope] = []
    private let capacity: Int                  // ~500
    private var dropEventEmitted = false       // 1회 발행 래치
    private(set) var dropEventCount = 0

    public init(capacity: Int = 500) { self.capacity = capacity }

    /// envelope 단위 enqueue. overflow 시 가장 오래된 envelope을 통째로 drop
    /// (byte 절단 불가능 → UTF-8 안전). 반환: 발행할 drop-mark(있으면).
    public func enqueue(_ env: WSEnvelope) -> WSEnvelope? {
        items.append(env)
        guard items.count > capacity else { return nil }
        items.removeFirst()                    // 메시지(envelope) 경계 drop
        if !dropEventEmitted {
            dropEventEmitted = true
            dropEventCount += 1
            return Self.makeDropMark()         // kind=error, code=BUFFER_OVERFLOW_DROPPED
        }
        return nil
    }

    public func drain(upTo n: Int) -> [WSEnvelope] {
        let taken = Array(items.prefix(n))
        items.removeFirst(min(n, items.count))
        if items.count < capacity { dropEventEmitted = false }   // 여유 회복 → 래치 리셋
        return taken
    }
}

5.3 SerialInputQueue + InputSink (origin별 2-executor, strict FIFO) + Utf8StreamAccumulator

구현 메모
아래 의사코드는 error code를 String 리터럴(onError("PTY_WRITE_FAILED") 등)로 표기하나, 구현 시 단일 enum(DaemonErrorCode)으로 통일하여 테스트의 .ptyWriteFailed case 비교(§3 Day5)와 정합시킨다. String↔enum 매핑 helper는 의사코드에서 생략. 
struct InputItem: Sendable { let bytes: [UInt8]; let isControl: Bool }

/// 각 sessionId마다 단일 AsyncStream + 단일 consumer. 단일 consumer가 각 item의
/// origin별 write를 await로 완결한 뒤 다음 item 처리 → strict FIFO. 동시 write 구조적 불가.
actor SerialInputQueue {
    private var continuations: [UUID: AsyncStream<InputItem>.Continuation] = [:]
    private var consumers: [UUID: Task<Void, Never>] = [:]

    func attach(sessionId: UUID, sink: InputSink) {
        guard consumers[sessionId] == nil else { return }   // 단일 consumer 보장
        let stream = AsyncStream<InputItem> { cont in continuations[sessionId] = cont }
        consumers[sessionId] = Task {
            for await item in stream { await sink.write(item) }  // await 완결 → 다음. 분리 Task 금지.
        }
    }
    func enqueue(_ item: InputItem, for sessionId: UUID) { continuations[sessionId]?.yield(item) }
    func detach(sessionId: UUID) {
        continuations[sessionId]?.finish(); continuations[sessionId] = nil
        consumers[sessionId]?.cancel(); consumers[sessionId] = nil
    }
}

protocol InputSink: Sendable { func write(_ item: InputItem) async }

/// .external — actor 소유 masterFD. 기존 PTYWriter.write 재사용(full-write + EAGAIN 루프).
/// raw Darwin.write 단발 호출 금지: masterFD가 O_NONBLOCK(PTYSpawner.swift:42)이라 EAGAIN/short-write 시 byte 유실.
actor ExternalSink: InputSink {
    let masterFD: Int32
    let onError: (String) -> Void   // PTY_WRITE_FAILED 발행(InternalSink.onUnsupported 와 대칭)
    func write(_ item: InputItem) async {
        do { try PTYWriter.write(masterFD, Data(item.bytes)) }   // full-write+EAGAIN, 데이터 손실 0
        catch { onError("PTY_WRITE_FAILED") }   // hard-failure(PTYError.fcntlFailed) 가시화 — silent drop 금지
    }
    // 주의(P4 한정 부채): PTYWriter는 EAGAIN 시 usleep(1ms) spin. PTY가 가득 차면 actor executor 점유.
    //   P4 데모 부하에서는 미발현. internal 실사용/고부하 wiring 시 DispatchIO write로 전환 검토(§9 Risks).
}

/// .internal — MainActor에서 libghostty 주입. printable만 지원, control은 미지원 신호.
@MainActor
final class InternalSink: InputSink {
    let surface: ghostty_surface_t
    let onUnsupported: (String) -> Void   // INTERNAL_CONTROL_INPUT_UNSUPPORTED 발행 콜백
    nonisolated func write(_ item: InputItem) async {
        await MainActor.run {
            if item.isControl {
                // C1: control byte(<0x20)는 libghostty가 keycode+mods로 자체 인코딩하므로 raw 주입 불가.
                //   ESC 다중 byte 역매핑도 불가. P4는 미지원 — 명시 신호.
                onUnsupported("INTERNAL_CONTROL_INPUT_UNSUPPORTED")
            } else {
                item.bytes.withUnsafeBytes { raw in
                    ghostty_surface_text(surface, raw.bindMemory(to: CChar.self).baseAddress, UInt(item.bytes.count))
                }
            }
        }
    }
}

/// 출력 chunk 경계 재조립. PTYReader는 lowWater 1(PTYReader.swift:34)이라 멀티바이트가 chunk 경계에서
/// 쪼개진다. 완성 UTF-8 prefix만 방출하고 미완 byte는 carry에 보관 → 다음 chunk와 결합.
/// Utf8BoundaryTruncator(grapheme truncate 유틸)는 재사용 불가.
struct Utf8StreamAccumulator {
    private var carry: [UInt8] = []
    /// 입력 chunk를 누적하여 완성 UTF-8 문자열을 방출. 미완 byte는 carry로 이월.
    /// carry가 상한(4 byte = 최대 UTF-8 시퀀스)을 넘으면 malformed로 간주, U+FFFD 치환 후 flush(무한 적체 방지).
    mutating func push(_ chunk: Data) -> String? {
        var buf = carry + Array(chunk)
        // buf 끝에서 불완전한 멀티바이트 시퀀스 길이를 계산하여 잘라 carry로 보관.
        let split = Self.completeUtf8PrefixLength(buf)
        let complete = Array(buf[..<split]); carry = Array(buf[split...])
        return complete.isEmpty ? nil : String(decoding: complete, as: UTF8.self)
    }
}

5.4 Session bind API + per-client lastSeq + disconnect 정리 (PTY 생존)

actor SessionBindRegistry {
    private var bindings: [UUID: UUID] = [:]    // clientId → sessionId
    private var lastSeq: [UUID: UInt64] = [:]   // clientId → 마지막 inbound seq

    func bind(clientId: UUID, sessionId: UUID) { bindings[clientId] = sessionId; lastSeq[clientId] = 0 }
    func boundSession(clientId: UUID) -> UUID? { bindings[clientId] }

    /// inbound envelope의 seq 모노토닉성 검증(§5.1). 역행/중복 시 throw → 호출부가 NON_MONOTONIC_SEQ 발행.
    func ingestInbound(clientId: UUID, env: WSEnvelope) throws {
        var s = lastSeq[clientId] ?? 0
        try validateMonotonic(env, lastSeq: &s)
        lastSeq[clientId] = s
    }

    /// client 비정상 종료 시. 매핑/lastSeq만 제거 — SessionManager.terminate 미호출(PTY 생존).
    func cleanup(clientId: UUID) { bindings[clientId] = nil; lastSeq[clientId] = nil }
}

6. ADR 운영화

ADR-I(cmux surface) 정합

ADR-I상 모든 surface는 SurfaceRegistry owner가 등록~release까지 보유하여 alive하다(SurfaceLifecycle.swift:18~46). P4의 internal printable 입력 주입은 이 invariant에 의존한다: 비-focus/inactive tab surface도 살아있으므로 ghostty_surface_text를 핸들로 호출 가능(C2). P4는 hibernate/pause 코드를 추가하지 않으며 ADR-I를 그대로 따른다.

R8(single-writer) 정책의 코드화

R8을 SerialInputQueue(§5.3)로 운영화한다: (1) 세션별 단일 consumer로 동시 write 2개 이상 구조 금지. (2) consumer가 각 write를 await로 완결 후 다음 → strict FIFO. (3) origin별 sink로 external/internal 비대칭 흡수. R8의 cross-origin(keystroke+WS) 시나리오는 external에 GUI keystroke 경로가 없어 P4 미검증임을 §8 A7에 명기한다.

ADR-P4-1 — Single-writer 큐: Option A 채택, sink 경계 유지로 Option B 이월

  • Decision: P4는 모든 입력을 SerialInputQueue로 funnel하는 Option A를 채택. internal 로컬 keystroke fast-path(Option B)는 후속 phase로 이월하되, InputSink 추상 경계를 깨끗이 유지하여 후속 phase에서 큐 재설계 없이 sink 교체만으로 전환 가능하게 한다.
  • Drivers: DR1, DR2.
  • Alternatives: Option B — internal keystroke 지연 0 이득이 P4 external 데모에서 미측정, generation/lock race 복잡도는 즉시 비용.
  • Why chosen: external 데모라 Option A의 keystroke-지연 비용이 P4에서 미발현. strict FIFO 정합 이득만 취함.
  • Consequences: internal 실사용 wiring 시 keystroke 지연이 비용 → 후속 phase에서 Option B 부분 전환 재평가(부채 명시).
  • Follow-ups: internal 로컬 입력 fast-path를 internal 출력 tap 확보 phase와 함께 결정.

ADR-P4-2 — seq/ackSeq 직렬화는 unconditional string

  • Decision: seq/ackSeq는 wire에서 항상 string. 내부는 UInt64.
  • Drivers: DR3.
  • Alternatives: number 직렬화 — 2^53 초과 손실로 거부.
  • Why chosen: 손실 없는 cross-runtime 호환.
  • Consequences: codec custom Codable. 라운드트립 테스트에 2^53+1 강제.
  • Follow-ups: P7 freeze 시 반영.

ADR-P4-3 — .internal 출력 스트리밍 P4 보류

  • Decision: P4는 .internal byte-faithful 출력 미지원(grid polling만). 요청 시 INTERNAL_OUTPUT_UNSUPPORTED. 데모/acceptance는 .external 기준.
  • Drivers: DR2(C4 — libghostty 출력 byte tap 부재, ghostty.h:1018~1027).
  • Alternatives: viewport read_text → 스트림 변환은 grid 스냅샷이라 ANSI/byte 정합 소실로 거부.
  • Why chosen: 런타임 콜백에 raw output tap 부재. P4 골격은 external-first 정당.
  • Consequences: GUI 실사용 출력 스트리밍은 P4 후 부채. Day 0 spike가 해결 옵션 열거 + P7 freeze 전 결정 게이트.
  • Follow-ups: docs/p4/c4-output-tap-spike.md의 결정 게이트.

ADR-P4-4 — .internal control byte 입력 P4 보류

  • Decision: P4는 .internal에 printable 입력만 주입. control byte(< 0x20, ESC 시퀀스) 수신 시 INTERNAL_CONTROL_INPUT_UNSUPPORTED. .external은 PTYWriter.write로 raw control 그대로 전달(제약 없음).
  • Drivers: C1 — control byte는 libghostty가 keycode+mods로 자체 인코딩(GhosttyTerminalView.swift:412~414), raw 주입 시 double-encode, ESC 다중 byte는 단일 keycode 역매핑 불가.
  • Alternatives: byte→keycode 역매핑 테이블 — ESC 시퀀스/조합키 환원 불가로 부분적·불완전. P4 골격에 부적합.
  • Why chosen: 미지원을 명시 신호로 정직하게 통지. external 데모는 영향 없음.
  • Consequences: 모바일에서 internal 세션에 Ctrl/방향키 전송은 후속 phase 과제(libghostty key 주입 설계 필요).
  • Follow-ups: internal 출력 tap phase와 함께 control 입력 경로 설계.
